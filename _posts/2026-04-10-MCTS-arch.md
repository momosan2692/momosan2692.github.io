## 1. Core Principle

> **Structure is the feature, not the model size.**  
> What a 1T-parameter monolithic LLM cannot do, four 7B agents with MCTS can.

A monolithic LLM fuses all knowledge into a single forward pass. That fusion is also destruction — once the weights mix cardiology with pharmacology with radiology, you cannot pull them apart again. The output has no seams. You cannot ask "which part of this answer came from drug interaction knowledge?" because there is no structural boundary where drug interaction knowledge lives.

The harness architecture **preserves those seams deliberately**:

- Each sub-agent owns its domain
- Each RAG retrieval is attributable
- Each confidence score is domain-specific
- Each MCTS node carries a provenance tag
- The final output has structure that mirrors the structure of the knowledge that produced it

This is why scaling the monolith doesn't solve the problem. A trillion-parameter model is *more* fused, not less. You get better answers on average but lose the ability to interrogate, override, or audit any specific reasoning step — which is exactly what regulated domains like medicine require.

### Four-Phase Breakdown

| Phase | Model role | Cost | Notes |
|---|---|---|---|
| **Selection** | None — pure UCB1 math | Zero | `argmax Q/N + c·√(ln Np/N)` |
| **Expansion** | Expander (creative, medium) | 1 call/iter | Temperature 0.9; generates N distinct steps |
| **Simulation** | Rollout (fast, cheap) | rollout_depth+1 calls | Hottest code path — keep depth ≤ 2 |
| **Evaluation** | Evaluator (strong) | 1 call total | Called once at end for final scoring |

### UCB1 Tuning

Classic `UCB_C = √2` works for games. LLM reasoning trees benefit from `UCB_C = 2.0–3.0` due to high variance in reasoning step quality. Tune empirically per task domain.

### Three Critical Implementation Notes

1. `best_child_exploit()` for final answer uses **pure exploitation** (no UCB exploration bonus). Never mix with UCB selection during search — it will return a high-variance unexplored branch as your final answer.

2. `rollout_depth` has a direct cost multiplier: 3 iterations × 3 rollout steps = 9 Rollout model calls per loop. Keep `rollout_depth ≤ 2` for interactive use.

3. The `state` field accumulates the **full chain**, not a delta. Token cost grows linearly with depth. Set `max_depth=4` as a hard ceiling.

## 3. Multi-Model Phase Assignment

The core AgentCore design pattern: each MCTS phase maps to a different model endpoint. No phase is hardcoded to a specific model.

```python
MODEL_MAP: dict[str, dict] = {
    "evaluator": {
        "model":    "qwen3:72b",
        "base_url": "http://localhost:11434/v1",
        "api_key":  "ollama",
    },
    "expander": {
        "model":    "qwen3:30b",
        "base_url": "http://localhost:11434/v1",
        "api_key":  "ollama",
    },
    "rollout": {
        "model":    "qwen3:7b",
        "base_url": "http://localhost:11434/v1",
        "api_key":  "ollama",
    },
}
```

Hot-swap any phase at runtime via `reassign_model(phase, model, base_url)`. This wires directly to AgentCore's `POST /admin/reassign` endpoint pattern.

**Cost asymmetry rationale:**

- Evaluator is called once → pay for quality
- Expander is called O(iterations) → pay for diversity
- Rollout is called O(iterations × depth) → pay for speed

Model capability and MCTS iteration budget are **not substitutes** — they multiply. Doubling iterations on a weak-model tree does not approach the quality of fewer iterations on a strong-model tree.

### Six ECO-World-Specific Challenges

| # | Challenge | Solution |
|---|---|---|
| ① | Continuous latent space — no discrete action enumeration | Progressive widening: `k(n) = c·nᵅ`, sample from `π(z)` |
| ② | Model error compounds per step | Hard cap rollout depth ≤ 5; use `V(z)` bootstrapping at leaf |
| ③ | Stochastic dynamics | Sample multiple `z'` per expansion; run expectimax averaging |
| ④ | Long horizon (decades for ecological policy) | Train `V(z)` to encode discounted future return — DreamerV3 approach |
| ⑤ | Non-stationary dynamics (climate shift, market regime) | Periodic WM fine-tuning; ensemble divergence detection |
| ⑥ | Multi-agent (species competition, economic actors) | `f(z, a_self, a_opponents) → z'` — bake opponent policy into dynamics |

**Practical decision rule:** For 30–90 day decision horizon → MCTS depth ≤ 5 + V(z) bootstrapping. For real-time per-step decisions → CEM or pure policy head `π(z)`.

| Timescale | Layer | Frequency |
|---|---|---|
| MCTS iterations | Outer search strategy — which branch? | Seconds to minutes |
| H module recurrence | Abstract planning per node | Milliseconds |
| L module steps | Fast local convergence inside H cycle | Microseconds |

### HRM Caveats for AgentCore

1. **Task-specific training required** — HRM has no cross-task transfer. One model per structured task type.
2. **Latent reasoning = non-interpretable** — critical compliance blocker for medical and trading system auditability.
3. **Fits as inner engine, not outer loop** — HRM-L can serve as fast rollout inside each MCTS node; MCTS provides the explainability wrapper.

## 6. HRM Claims vs Reality

The "27M parameters, 1000 examples" headline is significantly misleading on five levels:

| Claim | Reality | Implication |
|---|---|---|
| "1000 examples" | 1000 base × `num-aug` multiplier. Effective dataset is much larger. | True sample count is hidden behind augmentation |
| "Solves ARC, Sudoku, Maze" | Three separate training runs — no cross-task transfer shown | Architecture generalizes; weights do not |
| ARC-AGI 40.3% beats o3 | ARC Prize: "purely transductive" — test inputs must be in training set. ARC-AGI-2 result: **0% accuracy** | Not generalizing — memorizing structure |
| "Complex AI benchmarks" | All three tasks are grid-based, discrete, formally defined. No NLP, time-series, or financial tasks | Works for structured search; ECO-world needs validation |
| HRM beats same-size transformer | ARC Prize: same-size transformer comes within ~5pp without hyperparameter tuning | Loop count matters more than architecture |

**Bottom line:** Genuine architectural innovation — the H/L timescale separation and fixed-point gradient trick are real. But treat it as per-domain training cost, not a general plug-and-play reasoner.

### UCB-PUCT Formula

```
score(s,a) = Q(s,a)/N(s,a)  +  c · P(s,a) · √N(s) / (1 + N(s,a))
```

- `Q/N` — exploitation: known value
- `P(s,a)` — Watcher's prior: focuses budget on promising candidates
- `√N/(1+N)` — exploration: decreases as node is visited more

### Three Tree Regimes

| Regime | Behaviour |
|---|---|
| Within planning window | Tree grows each iteration — not fixed |
| Expected world step (action taken) | Re-root at executed child; all stats below survive (tree reuse) |
| Unexpected divergence | Watcher detects mismatch → signal reset → seed fresh tree with new candidates |

## 8. Model Capability Ceiling

The tree's shape is directly determined by model capability across four axes:

| Axis | Strong model | Weak model | Consequence |
|---|---|---|---|
| **Branch quality** | Diverse, valid, distinct candidates | Semantic duplicates, invalid branches | MCTS cannot find branches the Expander never generates |
| **Value accuracy** | Correct Q-signal → search in right direction | Inflated scores on weak branches | More compute makes it worse — deepens the wrong path |
| **Effective depth** | Coherent to depth 6–8 (32B+) | Degrades at depth 3–4 (7B) | `max_depth` must be calibrated per model, not as universal constant |
| **Halt calibration** | Knows when to stop → good exploration | Overconfident → prunes good paths early | Watcher scores must be calibrated probabilities, not raw logits |

> **Key law:** Model capability and MCTS iteration budget are not substitutes — they multiply.  
> Expander + Evaluator set the **quality ceiling**. Rollout sets the **speed floor**. Iterations operate within that envelope.

## 9. MCTS Output Modes

One MCTS run produces all three simultaneously. Different consumers use different parts:

| Mode | What | How | Used for |
|---|---|---|---|
| **A — best action** | `argmax Q(s,a)/N` at root | Returns single `a₁` | Game playing, real-time agents, ECO-world step execution |
| **B — full best path** | `s₀ → a₁ → b₁₁ → c₁₁₁` | Concatenate node `action` strings | CoT-MCTS reasoning trace; deepest leaf state = final answer; **medical audit trail** |
| **C — policy distribution** | `π(a|s) = N(s,a) / ΣN` | Probability over root actions | AlphaZero training signal; **Watcher training target** |

Mode B is the "LLM result" in the CoT-MCTS sense. The full path is the reasoning chain. The internal nodes are not scaffolding — they are the output.

### Why MCTS Beats HRM for Medical

| Requirement | MCTS | HRM |
|---|---|---|
| Explainable reasoning path | Mode B — every step is a text node | Latent only — no interpretable steps |
| Clinician override at any point | Override any node, re-plan from there | Single forward pass — no override points |
| Audit trail for regulation | Full path stored with provenance | No intermediate steps to audit |
| Domain attribution | Each node from specific sub-agent | Fused — no domain separation |
| Uncertainty signal | Visit distribution shows contested paths | Single output with no structural uncertainty |

**Latency split design:**

- Elective planning (treatment selection, med adjustment) → full MCTS, minutes budget
- Emergency triage → pre-computed policy from Mode C; fast Watcher only

### Agent Disagreement as a Signal

When two agents produce conflicting scores for the same candidate action, the combined UCB-PUCT prior drops and exploration increases on that node. The tree **automatically allocates more search budget to contested branches** — exactly where a clinician should also deliberate more. MCTS turns multi-agent disagreement into systematic deeper exploration rather than a silent averaged-out answer.

## 13. Alternative Aggregation Methods

Six meaningful alternatives — each handling the aggregation problem differently, each with a different tradeoff profile against MCTS for medical.

| Method | How it aggregates | Medical fit | Key gap vs MCTS |
|---|---|---|---|
| **① Debate** | Agents argue, judge LLM decides | Good — surfaces disagreement explicitly, trace readable | No systematic search; confident wrong agent can overrule judge |
| **② Blackboard** | Shared working memory; agents read/write/revise | Very good — natural for clinical notes; full provenance | No tree search; no systematic path ranking; convergence not guaranteed |
| **③ MoE router** | Gating function picks top-K agents; only selected run | Moderate — fast, cheap; routing errors are silent | No cross-agent synthesis; missed agents leave blind spots undetected |
| **④ Graph of Thoughts** | Like MCTS but allows node merges → DAG not tree | Strong — handles convergent evidence naturally | No UCB budget control; merge logic manually designed, not learned |
| **⑤ Reflexion** | Iterative self-critique + memory feedback | Moderate — good for refining single agent output | Single-agent by design; no multi-source provenance separation |
| **⑥ Ensemble vote** | Agents vote, weighted majority wins | Low — simple, fast; no reasoning path, no search | No audit trail; correct minority view can be outvoted silently |

### Combination Strategies (Non-Exclusive)

These methods are **composable**, not mutually exclusive:

- **Blackboard + MCTS** → shared state feeds the tree (most natural pairing). Blackboard = shared clinical state accumulator; MCTS = search layer on top. Different timescales, no conflict.

- **GoT + MCTS** → upgrade path when MCTS produces redundant branches. When two agents independently reach the same finding via different paths, GoT merges the nodes and increases combined evidence weight.

- **Debate → Expander input** → agents briefly challenge each other's top candidate before passing to MCTS. Richer, stress-tested candidate set. One round-trip latency added; significantly improves branch quality.

- **Reflexion inside each node** → agent self-critiques before passing reasoning step to next level. Improves node quality without touching the tree structure.

- **Ensemble vote → fast Watcher prior only** → cheap prior generation, never as final aggregator. Has no audit trail and can silently outvote a correct minority — unacceptable at output layer.

## 14. Implementation Reference

### Core Python — MCTSNode

```python
from dataclasses import dataclass, field
from typing import Optional
import math

UCB_C = 1.414  # tune to 2.0–3.0 for LLM reasoning

@dataclass
class MCTSNode:
    state:    str                          # accumulated reasoning chain
    action:   str = ""                     # single step that led here
    parent:   Optional['MCTSNode'] = None
    children: list['MCTSNode'] = field(default_factory=list)
    visits:   int   = 0
    value:    float = 0.0

    @property
    def ucb1(self) -> float:
        if self.visits == 0:
            return float("inf")
        if self.parent is None or self.parent.visits == 0:
            return self.value / self.visits
        return (self.value / self.visits) + UCB_C * math.sqrt(
            math.log(self.parent.visits) / self.visits
        )

    def best_child_ucb(self):      # use during search
        return max(self.children, key=lambda n: n.ucb1)

    def best_child_exploit(self):  # use for final answer (Mode A)
        return max(self.children, key=lambda n: n.value / max(n.visits, 1))
```

### UCB-PUCT with Watcher Prior

```python
def ucb_puct(node, prior: float, c: float = 1.414) -> float:
    """UCB-PUCT: standard UCB1 augmented with prior from fast Watcher."""
    if node.visits == 0:
        return float("inf")
    q = node.value / node.visits
    n_parent = node.parent.visits if node.parent else 1
    exploration = c * prior * math.sqrt(n_parent) / (1 + node.visits)
    return q + exploration
```

### Mode C — Policy Distribution (Watcher Training Target)

def policy_distribution(root: MCTSNode) -> dict[str, float]:
    """Mode C: visit-weighted distribution over root actions."""
    total = sum(c.visits for c in root.children)
    if total == 0:
        return {}
    return {c.action: c.visits / total for c in root.children}

## 15. Decision Guide

### Choose MCTS when:

- Sequential decision making with branching alternatives
- Explainability and audit trail required (medical, legal, financial regulated)
- Human override at intermediate steps required
- Multi-agent disagreement should be surfaced, not averaged
- Planning horizon allows seconds to minutes of compute
- Domain knowledge comes from multiple distinct RAG+LLM sources

### Choose HRM when:

- Grid-based, discrete, formally structured task (puzzle, maze, grid planning)
- No interpretability requirement
- Single domain (one training run per task)
- Latency is critical and task is well-defined
- Edge device deployment (27M parameters fits anywhere)

### Choose Debate when:

- Fewer than 4 agents, small action space
- Disagreement surfacing is primary goal, not deep search
- Latency budget is tight

### Choose Blackboard when:

- Agents need to build on each other's findings incrementally
- No ranking of final paths needed — convergence is sufficient
- Works best combined with MCTS on top

### Choose GoT when:

- Multiple reasoning paths legitimately converge on the same evidence
- You've already validated MCTS and find branch redundancy
- GoT is an upgrade path from MCTS, not a replacement

### Emergency vs Elective Medical Split:

```
Elective planning (treatment selection, medication adjustment):
  → Full MCTS + human gate  →  minutes budget

Emergency triage:
  → Pre-computed Mode C policy from MCTS offline training
  → Fast Watcher prior only at inference
  → Human gate still mandatory
```

## Summary

This document covers the complete architectural reasoning chain developed across one research session:

1. MCTS is the correct aggregation mechanism for multi-domain knowledge integration because it preserves seams between knowledge sources as auditable tree nodes.
2. Model capability sets the quality ceiling — iteration budget operates within that envelope.
3. The fast Watcher (Policy Prior pattern) prevents wasted iterations and detects world divergence for tree reset.
4. HRM's latent reasoning disqualifies it from medical use; it belongs as an inner rollout engine inside MCTS nodes, not as the outer reasoning framework.
5. Sub-agents (RAG+LLM per domain) are knowledge producers. MCTS is the decision integrator. They never directly touch each other's internals.
6. Agent disagreement becomes a MCTS exploration signal — the most underappreciated emergent property of this architecture.
7. The harness principle: structure is the feature. Four 7B agents with MCTS produce capabilities a 1T-parameter monolith structurally cannot — traceable provenance, partial override, calibrated per-domain confidence, and systematic disagreement surfacing.
