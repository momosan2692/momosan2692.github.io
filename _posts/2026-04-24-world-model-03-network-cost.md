# 戴著鐐銬跳舞——禁令下的萬卡互連黑科技，與訓練推論成本的壓縮終局

> 世界模型系列 #3 · April 2026  
> Covers: RoCE v2 優化、萬卡通訊黑科技、TP+PP+DP 三級並行、帶內遙測、訓練成本、推論成本壓縮路線圖

---

## 1. 為什麼網路是萬卡叢集的死穴

世界模型的訓練不是一個「買夠多 GPU、插上電就能跑」的工程問題。當訓練規模擴展到萬張 GPU 時，制約系統效率的瓶頸從**算力**轉移到了**通訊**。

理解這個轉移，需要看一個關鍵的數學事實。

在分散式訓練中，最常用的梯度同步操作是 **All-Reduce**——讓所有 GPU 上的梯度取平均，確保每張卡上的模型權重保持一致。對於 $$N$$ 張 GPU 的 Ring All-Reduce，每張卡需要傳輸的數據量是：

$$\text{通訊量} = 2 \times \frac{N-1}{N} \times M \approx 2M \quad (N \gg 1)$$

其中 $$M$$ 是模型參數的總大小（以字節計）。對於一個 100B 參數的模型（BF16 精度），$$M \approx 200$$ GB。這意味著每一步訓練，每張卡都需要傳輸 **~400 GB 的梯度數據**。

如果叢集的跨節點頻寬是 400 Gb/s（典型的 InfiniBand HDR），那麼每步的梯度同步需要至少 **8 秒**。在 GPU 算力提升到每步計算只需要 **0.5 秒**的情況下，GPU 有 94% 的時間在等待網路——這就是「通訊牆（Communication Wall）」的直觀含義。

## 2. 硬體現實：H20 的「畸形」設計

NVIDIA 在設計 H20 時，為了符合美國商務部的出口管制門檻，對計算能力進行了大幅削減，但有意保留了高頻寬互連能力。

| 規格 | H100 SXM5 | H20 | 比例 |
|---|---|---|---|
| BF16 Tensor Core TFLOPS | 1,979 | ~296 | ~15% |
| HBM3 頻寬 | 3.35 TB/s | 4.0 TB/s | 119%（H20 反而更高） |
| NVLink 頻寬（節點內） | 900 GB/s | 900 GB/s | 100% |
| NVLink 版本 | NVLink 4.0 | NVLink 4.0 | 相同 |
| HBM 容量 | 80 GB | 96 GB | 120% |

這個設計揭示了 NVIDIA 的精確算計：H20 的節點**內部**互連能力與 H100 完全相同，但節點之間的跨機器通訊所需的高端 InfiniBand 交換機（Quantum-2）同樣受到禁令限制。結果是：8 張 H20 組成的單機節點效率接近 H100 節點，但**跨節點的萬卡叢集效率**被系統性壓制。

大陸廠商的應對邏輯因此變成：**盡一切可能讓計算發生在 NVLink 覆蓋的節點內，只讓壓縮到極致的少量數據跨節點傳輸。**

### 國產替代：華為 Ascend 910C 的真實差距

| 指標 | H20 | 華為 Ascend 910C | 說明 |
|---|---|---|---|
| BF16 算力（理論） | ~296 TFLOPS | ~256 TFLOPS | 相近 |
| HBM 頻寬 | 4.0 TB/s | 3.2 TB/s | H20 略優 |
| 節點內互連頻寬 | 900 GB/s（NVLink） | ~600 GB/s（HCCS） | H20 優 |
| 軟體生態（框架支援） | CUDA 完整支援 | CANN，PyTorch 約 70–80% | H20 大幅領先 |
| 實際訓練 MFU（LLM） | ~45–55% | ~30–40% | 因生態不成熟，差距大於理論 |

910C 的**理論算力紙面上不差**，但因為 CANN 軟體棧的算子覆蓋不完整、與 PyTorch/JAX 的整合仍有缺口，**實際可用效率比 H20 低 20–30%**。智譜 AI 等國家隊能接受這個代差；MiniMax 等商業化速度優先的公司目前仍優先消耗存量的 A100/H800。

## 3. 軟體優化的五個層次

既然硬體存在先天代差，大陸工程師就只能在軟體棧上尋求補償。這是一場「用算法對抗物理定律」的工程戰。

### 層次一：通訊與計算的深度掩蓋（Zero Bubble Pipeline）

最直觀的優化：既然通訊慢，就讓通訊「隱形」。

進階的流水線並行調度（Advanced Pipeline Parallelism）將模型拆解為更細小的微批次（Micro-batch），並精確安排計算與通訊的時間順序：

```
GPU A 計算第 1 層  →  GPU A 傳輸第 1 層梯度
                   →  同時，GPU A 開始計算第 2 層
GPU B 接收梯度     →  GPU B 立刻開始計算對應層
```

透過精確的時間對齊，讓 GPU 在傳輸當前層梯度的同時，已經開始計算下一層。目標是達到**「零氣泡（Zero Bubble）」**調度——GPU 幾乎感覺不到網路延遲的存在。

### 層次二：拓撲感知的三級並行（TP + PP + DP）

這是大陸萬卡叢集最核心的調度策略，其本質是**讓通訊量與頻寬精確匹配**。

不同的並行策略，通訊模式差異巨大：

**Tensor Parallelism（TP，張量並行）**：將單個矩陣乘法切分到多張 GPU 上同步計算。每一步都需要 All-Reduce，且通訊量與矩陣的行列維度成正比——通訊量極大，延遲要求極嚴。這類計算**必須鎖定在 NVLink 覆蓋的節點內（900 GB/s）**，否則效率會崩潰。

**Pipeline Parallelism（PP，流水線並行）**：將模型的不同層切分到不同 GPU 上，每一層的輸出作為下一層的輸入。通訊量中等（只傳輸激活值，不傳梯度），延遲可容忍毫秒級。這類計算適合**鎖定在同一機架內（RoCE 頂層交換機，~100 GB/s）**。

**Data Parallelism（DP，數據並行）**：在多個 GPU 上訓練不同的數據批次，定期同步梯度。通訊量相對較小（只需梯度同步），且可以使用梯度壓縮進一步縮減。這類計算適合**跨機架（RoCE 核心層，~25–40 GB/s）**。

將這個映射可視化：

```
節點內（NVLink 900 GB/s） ←→  Tensor Parallel（TP）
機架內（RoCE 100 GB/s）   ←→  Pipeline Parallel（PP）
跨機架（RoCE 25 GB/s）    ←→  Data Parallel（DP）
```

調度軟體（如字節跳動自研的 BytePS、阿里的 ACCL）會根據物理拓撲圖自動決定每一種並行的 group 分配，確保最頻繁的數據交換永遠發生在頻寬最高的地方。這是「讓計算適應網路，而非讓網路適應計算」的設計哲學。

### 層次三：突破 ECMP 的自適應路由

傳統的 ECMP（等價多路徑路由）基於流的五元組（src IP, dst IP, src port, dst port, protocol）進行 Hash 分流。AI 訓練流量的特點是**少數幾個極大的「象流（Elephant Flow）」**——這些流的 Hash 結果通常集中在少數幾條路徑上，導致這幾條路徑塞爆，而其他路徑空閒。

大陸廠商開發的**自適應路由（Adaptive Routing, AR）**打破了傳統 ECMP 的流級一致性限制，在**封包（Packet）級別**進行動態負載均衡：

1. 交換機即時偵測所有出口隊列的深度
2. 如果某條路徑的隊列超過閾值，將這個流的**後續封包**切換到更空閒的路徑
3. 接收端由軟體協議棧負責封包的重排序（Reordering）

這將網路利用率從 ECMP 的 **55–65%** 提升到 **85–90%** 以上，等效於在不增加任何硬體的情況下，讓整個叢集多出了 30% 的有效頻寬。

### 層次四：RoCE v2 的擁塞控制——解決 PFC 死鎖

RoCE v2（RDMA over Converged Ethernet）依賴無損網路（Lossless Network），使用 PFC（Priority Flow Control）機制：當下游設備的緩衝區即將溢出時，向上游發送暫停幀，讓上游停止發送。

在萬卡叢集中，這個機制極易引發**連鎖死鎖（PFC Storm）**：A 暫停 B，B 轉而暫停 C，C 又暫停 D——直到整個通訊路徑完全凍結。

大陸廠商的應對是在 DCQCN 基礎上開發更激進的擁塞控制算法，核心參數配置如下：

**交換機側：**
- ECN 標記門檻設在佇列深度 **20–30%**（遠低於傳統建議的 50%），提早發出擁塞信號
- PFC Headroom Buffer 精確計算：`Headroom = RTT × Line Rate ÷ Cell Size + Safety Margin`，防止因緩衝區不足導致的假性丟包

**網卡側：**
- Rate Decrease Factor（$$R_{ai}$$）調整為更激進的 **0.1–0.2**（原預設 0.5），讓發送速率更快收斂到網路容量
- 啟用 RNR（Receiver Not Ready）NAK 保護，防止因慢消費端堆積觸發的連鎖重傳

**整體目標**：在萬卡叢集中把 Packet Drop Rate 控制在 **< 0.001%**，讓 RDMA 重傳觸發率低於訓練吞吐的雜訊水準。

### 層次五：帶內網路遙測（INT）——一秒內定位故障光纖

在萬卡叢集中，一個「灰度故障」（如某根光纖折損導致的 0.01% 丟包率）會讓整個叢集的訓練效率悄悄下降 15–20%，且很難定位。傳統的 SNMP 輪詢監控有數十秒到分鐘的延遲，完全無法應對這個場景。

**In-band Network Telemetry（INT，帶內網路遙測）**的解決方案是：在每一個數據封包的頭部插入一個時戳欄位，要求封包經過的每一台交換機在轉發時，將當前的時延、隊列深度、出口丟包計數寫入這個欄位。封包到達接收端時，攜帶了沿途每一跳的精確遙測數據。

效果是：當萬卡訓練出現效率異常時，AI 調度器能在**一秒內**分析出「第 47 機架 / 3 號交換機 / 下行 12 號端口」的丟包率異常，立即隔離這條路徑，讓訓練任務自動切換到備用路徑繼續執行，而不需要人工介入或停機排查。

INT 的規範由 P4.org 在 2015–2016 年提出，2018 年後進入商用交換機（Barefoot Tofino、Intel）。但在萬卡 AI 叢集的具體應用——尤其是「閉環自動故障隔離」的全自動化——是 2023–2024 年才在字節跳動、Meta、阿里的內部叢集中實驗性部署的前沿實踐。

## 4. 萬卡優化的實際效益

上述五個層次的優化，並非相互獨立，而是可以疊加使用。根據 Megatron-LM、DeepSpeed、BytePS 等公開論文與實測數據的綜合估計：

| 優化技術 | 訓練吞吐量提升 | 適用規模 |
|---|---|---|
| Flash Attention v2/v3 | +2–4× 速度 | 所有規模 |
| Zero Bubble Pipeline | +15–30% MFU | 百億以上 |
| TP+PP+DP 拓撲映射 | +20–35% MFU | 萬卡叢集 |
| 自適應路由（vs ECMP） | 網路利用率 +30% | 跨機架通訊 |
| MoE 稀疏激活（1/8） | 等效算力 +4–6× | 千億以上 |
| INT 故障隔離 | 減少停機時間 ~60% | 萬卡持續訓練 |

**疊加效益**：未優化的萬卡叢集 MFU 通常在 30–40%（即算力的 60–70% 都在等待通訊或空轉）。全棧優化後，MFU 可提升到 **55–65%**，等效於在不增加任何硬體的情況下，讓整個叢集「多出」了將近一倍的有效算力。

這解釋了為什麼大陸廠商能用 H20 + A100 混合叢集，跑出接近 H100 純叢集的訓練效果——不是靠硬體，而是靠頂尖工程師「把每一個比特的傳輸路徑都算到極致」。

## 5. 子話題：從億元一次到一毛錢一秒——成本壓縮的終局

萬卡叢集解決了「如何訓練」的問題，但還有一個更根本的商業問題：**訓練了之後，怎麼讓推論便宜到進入普通人的手機？**

**訓練只發生一次，推論每天發生億次。** 長遠來看，推論成本才是決定「世界模型能否成為基礎設施」的關鍵門檻。

### 5.1 訓練成本的三個壓縮維度

**算法效率（最根本的槓桿）**

| 技術 | 成本影響 | 成熟度 |
|---|---|---|
| MoE 稀疏激活 | 等效訓練成本降至 1/4–1/6 | ✅ 已商用（DeepSeek-V3） |
| Flow Matching 取代 DDPM | 訓練步數 −60–70% | ✅ 已商用（Vidu、SD3） |
| 課程學習 | 減少無效迭代 ~30% | ✅ 業界標準 |
| 合成數據自舉 | 減少真實數據成本 | ⚠️ 品質風險仍存 |

**硬體利用率（工程側槓桿）**：MFU 從 30% 提升到 60%，等效算力翻倍，訓練成本直接減半——這是上一節萬卡優化的直接財務含義。

**「一次訓練、多次蒸餾」的攤薄策略**：訓練一個 100B 的大模型，再蒸餾出 7B / 1.5B 用於部署。訓練成本被攤薄到整個產品線。智譜的 GLM 系列、MiniMax 的 M 系列都是這個邏輯的實踐。

### 5.2 推論成本的四層壓縮

推論成本的壓縮有四個層次，從最普及到最前沿依次排列：

**第一層：量化（Quantization）**

```
FP32 → BF16 → FP8 → INT8 → INT4 → Binary（BitNet b1.58）
每降一級，顯存減半，速度約提升 1.5–2×
代價：精度損失，需 QAT（量化感知訓練）補償
```

**BitNet b1.58** 是這個方向的極端情境——模型權重只有 -1、0、+1 三個值，所有乘法運算退化為加法，理論上可以在**不需要乘法器的 ASIC** 上運行，直接繞過對 GPU 的依賴。對大陸的算力禁令而言，這是一個具有戰略意義的長期方向：如果未來的世界模型推論可以在專用 ASIC 或 RISC-V 架構的邊緣設備上完成，禁令對算力的壓制將從根本上失效。

**第二層：KV Cache 壓縮**

視訊世界模型的序列極長（60 秒的視訊可能產生 100K+ tokens），KV Cache 的顯存佔用會迅速達到 80GB HBM 的上限。主流壓縮技術包括：

- **GQA（Grouped Query Attention）**：Llama 3、Qwen 2 等模型的標配，KV Cache 縮減 4–8×
- **Sliding Window Attention**：只保留最近的 Token 窗口，適合即時推論場景
- **TurboQuant 類技術**：對 KV Cache 本身進行動態量化，進一步壓縮顯存佔用

**第三層：投機解碼（Speculative Decoding）**

用一個小模型（Draft Model）先快速生成後續 N 個 Token 的候選序列，再用大模型對這個序列進行**一次性批量驗證**。驗證比逐 Token 生成快 5–10×，且輸出品質與原模型完全一致（接受率高的情況下）。

整體推論速度提升 **2–3×**，且這個優化對模型架構完全透明——任何模型都可以在不修改權重的情況下受益。這是目前各大 API 廠商壓縮延遲的主流手段之一。

**第四層：端側部署（Edge Inference）——成本壓縮的終局形態**

推論完全發生在使用者的設備上，雲端只負責訓練。

```
雲端訓練大模型（一次成本，攤薄到所有用戶）
         ↓ 蒸餾 + 剪枝
設備端小模型（手機 NPU / Apple Silicon ANE）
         ↓ INT4/INT8 量化
完全本地推論，邊際成本 = 電費（~$0.0001 / 次）
```

對 Apple Silicon 生態（ANE + MLX 框架）而言，這個趨勢已經在發生——M4 Pro 的 ANE 算力達到 38 TOPS，理論上可以在本地即時運行 7B 參數量化後的小模型，推論成本接近零。

### 5.3 成本曲線：Wright's Law 在 AI 推論上的應用

視訊生成（10 秒 720P）的單次 API 成本走勢，清晰地呈現了指數級下降曲線：

| 時間 | 系統 | 單次成本（估算） |
|---|---|---|
| 2023 | Runway Gen-2 | ~$5–8 |
| 2024 | Sora 早期（估算） | ~$0.5–2 |
| 2025 | Vidu / Hailuo 初期 | ~$0.1–0.5 |
| 2026 Q1 | MiniMax Hailuo-2.3-Fast | ~$0.05–0.15 |
| 2027（預測） | Flow Matching + INT8 全棧 | ~$0.01 |
| 2028–2029（預測） | 端側部署普及後 | ~$0.001 或免費 |

這個曲線符合 **Wright's Law**：每當累計生成量翻倍，單位成本下降約 20–30%。驅動力是算法優化而非製程進步——這與半導體的 Moore's Law 性質類似，但速度更快，且主要取決於工程人才密度而非資本投入。

### 5.4 商業含義：毛利率擴張的結構性邏輯

理解了成本曲線，就能理解大陸 AI API 市場看似矛盾的現象：

智譜 AI 在 2026 年 Q1 將 API 定價提高 83%，但呼叫量同期增長 400%，MaaS 平台毛利率從低個位數擴張到 18.9%。

這不是矛盾，而是**供給側成本下降速度快於定價下降速度**的必然結果。量化後：

```
定價上漲 83% → 每次呼叫收入 +83%
優化後單次成本 → 下降約 60–70%（Flash Attention + Flow Matching 疊加）
淨效果 → 單次毛利率大幅擴張
同時，呼叫量 +400% → 總毛利潤爆發式增長
```

MiniMax 走相反的定價策略（極致低價），但其 API 平台毛利率達到 69%，原因是其推理引擎優化能力在業界最強，同樣的成本壓縮讓它在低定價下仍然保持了豐厚的利潤空間。

## 6. 當算力禁令鬆動時，哪些技術會消失？

這是一個值得思考的反事實問題。

如果 NVIDIA H100 / H200 的出口限制明天全部解除，大陸廠商從禁令壓力下開發出的這些軟體優化技術，哪些會隨著「不再需要」而被放棄，哪些會因為其固有價值而被保留？

**可能被棄用（因為代價大於收益）**：
- 極激進的 RoCE PFC 參數調優——有了 InfiniBand，回到有損網路的麻煩可以直接規避
- 複雜的多路徑封包級自適應路由——InfiniBand 的硬體自適應路由直接解決問題

**會被永久保留（因為效益與硬體無關）**：
- Flash Attention：純算法優化，對所有硬體都有效
- Flow Matching：訓練效率的本質提升，不依賴硬體
- MoE 稀疏激活：讓模型更大但計算量不等比增長，是未來模型架構的基礎
- INT 帶內遙測：在任何規模的叢集中，毫秒級故障定位都是剛需
- 投機解碼：推論加速的通用技術，與硬體無關
- 端側量化部署：成本壓縮的終局方向，算力充裕也不改變這個趨勢

最具深意的結論是：**禁令加速了大陸 AI 工程師在軟體優化上的深度投入，而這些投入中最有價值的部分，與算力本身無關。** 即使禁令解除，這批工程能力也已經成為不可逆的技術資產。這是「戴著鐐銬跳舞」最後反轉的地方——鐐銬鍛煉出的肌肉，在鐐銬解除後依然存在。

## 7. 世界模型系列總結

三篇文章走過了一條完整的敘事弧線：

**第一篇**確立了世界模型的定義與本質挑戰——時空一致性、交互性、因果推理，以及「視訊模型 ≈ 世界模型」這個等式的成立條件與剩餘缺口。

**第二篇**深入了架構競爭的數學底層——U-ViT 的跳躍連接如何保護長視訊的物理一致性，DiT 的 adaLN-Zero 如何支撐超大規模訓練，CogVideoX 的 3D-VAE 如何用壓縮換效率，以及各種推理優化技術如何將 API 成本從美元級壓縮到分錢級。

**第三篇**呈現了算力受限下的工程現實——RoCE 的死穴與補丁、三級並行對網路拓撲的精確映射、INT 遙測的閉環故障隔離，以及最終的成本壓縮終局：Wright's Law 驅動下，世界模型的推論成本正在走向接近零的邊際成本。

世界模型距離真正意義上的實現——能夠理解、預測、並主動介入物理世界——仍然有「交互性」和「因果不可逆性」這兩道尚未跨越的技術鴻溝。但從訓練效率、推論成本、硬體普及的趨勢來看，這個技術的商業化曲線正在急劇陡峭。

什麼時候世界模型會成為一個我們習以為常的基礎設施，就像今天的 LLM 一樣？答案可能比大多數人預期的，要早得多。
