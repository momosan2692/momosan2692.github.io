# DeepSeek 技術演進報告
## 從架構創新到推理突破的完整技術發展脈絡

---

## 摘要

DeepSeek 作為中國 AI 領域的重要參與者,在大型語言模型(LLM)架構創新與效能優化方面取得了顯著成果。本報告深入分析 DeepSeek 自 2024 年初至今的技術演進,涵蓋其核心架構創新、訓練策略以及最新的 mHC(Manifold-Constrained Hyper-Connections)技術。

DeepSeek 的技術路線展現了一個清晰的發展邏輯:從混合專家系統(MoE)的優化,到注意力機制的革新,再到推理能力的突破,最終達到架構層面的根本性改進。這些創新不僅在理論上具有突破性,更在實踐中證明了以更低成本實現世界級 AI 性能的可能性。

## 一、引言:未來大模型的演進方向

### 1.1 核心問題

當前大型語言模型的發展面臨一個關鍵問題:**除了單純堆疊參數之外,模型架構本身是否還有進化空間?** 

這個問題的重要性在於:
- **參數規模的邊際效益遞減**:單純增加參數帶來的性能提升越來越小
- **訓練成本急劇上升**:GPT-4 的訓練成本估計超過 1 億美元
- **硬體資源限制**:並非所有團隊都能獲得最先進的 GPU 集群
- **能源與環境考量**:大規模訓練的能源消耗成為關注焦點

### 1.2 DeepSeek 的答案

DeepSeek 的技術發展歷程為這個問題提供了肯定的答案。從 DeepSeekMoE 的專家特化策略,到 Multi-head Latent Attention(MLA)的 KV cache 優化,再到 DeepSeek-R1 的純強化學習推理,以及最新的 mHC 架構創新,每一步都展現了**在保持訓練穩定性的同時提升模型能力**的可能性。

關鍵創新點:
- **效率優先**:以更少的激活參數達到更好的性能
- **架構創新**:不依賴最新硬體,通過算法優化突破限制
- **開放精神**:開源模型權重,推動整個領域發展
- **成本控制**:DeepSeek-V3 的訓練成本僅約 600 萬美元

## 二、DeepSeek 技術演進時間線

### 2.1 DeepSeekMoE: 專家特化的混合架構 (2024年1月)

**論文**: DeepSeekMoE: Towards Ultimate Expert Specialization in Mixture-of-Experts Language Models  
**arXiv**: 2401.06066

#### 核心創新

**1. 細粒度專家分割 (Fine-grained Expert Segmentation)**

傳統 MoE 架構存在專家知識冗餘問題。DeepSeekMoE 的創新在於:
- 將傳統的 N 個專家細分為 mN 個更小的專家
- 從中激活 mK 個專家,實現更靈活的專家組合
- 透過增加專家特化程度,減少知識重疊

**2. 共享專家隔離 (Shared Expert Isolation)**

- 設置 Ks 個共享專家來捕獲通用知識
- 減少路由專家中的冗餘
- 確保基礎能力的穩定性

**3. 性能突破**

實驗結果顯示:
- DeepSeekMoE 2B(總參數 16.4B,激活 2.8B)達到與 GShard 2.9B 相當的性能
- 激活參數比例僅為 18%,遠低於傳統 MoE 架構
- 在相同激活參數下,性能顯著優於 Mixtral 等競品

#### 技術細節

**專家配置**:
```
總專家數 = mN (細粒度專家) + Ks (共享專家)
每個 token 激活 = mK (路由專家) + Ks (共享專家)
```

**負載平衡策略**:
- 設備親和性損失 (Device Affinity Loss):確保專家在設備間均勻分佈
- 專家親和性損失 (Expert Affinity Loss):防止路由崩潰

#### 影響與意義

DeepSeekMoE 為後續所有模型奠定了基礎,證明了通過架構創新可以在不增加計算成本的情況下大幅提升模型容量。這一創新使得 DeepSeek 能夠在有限的硬體資源下(H800 GPU)訓練出世界級的大模型。

### 2.2 DeepSeek-V2: 注意力機制的革新 (2024年5月)

**論文**: DeepSeek-V2: A Strong, Economical, and Efficient Mixture-of-Experts Language Model  
**arXiv**: 2405.04434

#### 核心創新

**1. Multi-head Latent Attention (MLA)**

這是 DeepSeek-V2 最重要的創新,解決了 Transformer 架構的 KV cache 瓶頸問題。

**問題背景**:
- 傳統 Multi-Head Attention (MHA) 需要為每個 token 存儲完整的 Key-Value cache
- 隨著上下文窗口增長,記憶體需求呈線性增長
- 在長上下文場景下,KV cache 成為推理的主要瓶頸

**MLA 解決方案**:
- 將 Query、Key、Value 投影到低維潛在空間
- 使用低秩聯合壓縮 (Low-rank Joint Compression)
- 解耦旋轉位置編碼 (Decoupled Rotary Position Embedding, RoPE)

**效果**:
- KV cache 記憶體需求減少 **93%**
- 上下文窗口從 4K 擴展到 **128K tokens**
- 相當於可以處理約 10 萬字,接近一本《哈利波特》的長度

**技術原理**:

傳統 MHA:
```
每個 head: Q, K, V ∈ R^(d_model)
KV cache 大小: 2 × num_heads × d_head × seq_len
```

MLA:
```
壓縮: K, V → K_latent, V_latent ∈ R^(d_latent)
存儲: 只需存儲低維潛在向量
推理時: 再投影回原始維度
KV cache 大小: 大幅減少至原來的 ~7%
```

**2. DeepSeekMoE 架構優化**

在 DeepSeekMoE 基礎上進一步改進:
- 引入通訊平衡損失 (Communication Balance Loss)
- 確保多輪通訊在設備間均衡分佈
- 減少專家間通訊開銷

#### 模型規格

- **總參數**: 236B
- **激活參數**: 21B per token
- **上下文長度**: 128K tokens
- **硬體**: H800 GPU 集群

#### 訓練效率

DeepSeek-V2 在訓練效率上取得突破:
- API 定價: 2 RMB / 百萬輸出 tokens
- 比競品便宜數倍
- 在 University of Waterloo Tiger Lab 排行榜上位列第 7

#### 技術挑戰與解決

**挑戰 1: MLA 的實作複雜度**
- 需要自定義 CUDA kernel
- 精心設計的記憶體管理策略

**挑戰 2: MoE 的通訊開銷**
- 專家數量增加導致通訊成本上升
- 通過通訊平衡損失緩解

### 2.3 DeepSeek-V3: 大規模稀疏架構的成熟 (2024年12月)

**論文**: DeepSeek-V3 Technical Report  
**arXiv**: 2412.19437

#### 核心創新

**1. 無輔助損失的負載平衡策略 (Auxiliary-Loss-Free Load Balancing)**

**背景問題**:
- 傳統 MoE 使用輔助損失來平衡專家負載
- 需要調整超參數,增加訓練複雜度
- 輔助損失可能影響語言模型的主要訓練目標

**DeepSeek-V3 解決方案**:
- 每個專家維護一個偏置值 (bias value)
- 過度使用時偏置降低,使用不足時偏置提升
- 偏置值只影響 top-k 選擇,不影響實際的門控 logits
- 避免了額外的梯度計算和超參數調整

**2. 序列級輔助損失 (Sequence-wise Auxiliary Loss)**

為避免批次內的病理情況(pathological cases):
- 問題: 雖然整個批次在專家間平衡,但單個序列可能全用專家 X,另一序列全用專家 Y
- 解決: 在序列層面也施加平衡約束

**3. 多 Token 預測訓練目標 (Multi-Token Prediction, MTP)**

除了預測下一個 token,同時預測後續多個 tokens:
- 強化模型的長期規劃能力
- 提升整體性能
- 為推理任務打下基礎

#### 模型規格

- **總參數**: 671B
- **激活參數**: 37B per token
- **上下文長度**: 128K tokens
- **預訓練數據**: 14.8 trillion tokens

#### 訓練成本突破

這是 DeepSeek-V3 最令人矚目的成就:
- **訓練成本**: 僅約 **600 萬美元**
- **GPU 時數**: 2.788M H800 GPU hours
- **對比**: GPT-4 訓練成本估計超過 1 億美元
- **硬體**: 使用出口管制的 H800(非最先進的 H100)

#### 性能表現

- 在開源模型中表現最佳
- 與頂級閉源模型(如 GPT-4、Claude)性能相當
- 在多個基準測試中取得突破性成績

#### 基礎設施創新

**1. FP8 混合精度訓練**
- 使用 8-bit 浮點數進行部分計算
- 在保持精度的同時減少記憶體佔用和計算時間

**2. DualPipe 調度**
- 精心設計的流水線並行策略
- 隱藏通訊開銷
- 克服 H800 互連速度限制

**3. PTX 指令級優化**
- 直接使用 PTX (Parallel Thread Execution) 指令
- Warp 特化 (Warp Specialization)
- 最大化硬體利用率

### 2.4 DeepSeek-R1: 純強化學習的推理突破 (2025年1月)

**論文**: DeepSeek-R1: Incentivizing Reasoning Capability in LLMs via Reinforcement Learning  
**發布日期**: 2025年1月20日

#### 核心創新

**1. DeepSeek-R1-Zero: 純 RL 推理模型**

這是開源領域的首次嘗試:證明模型可以**僅通過強化學習**學會推理,無需監督微調(SFT)或學習的獎勵模型。

**關鍵特點**:
- 從 DeepSeek-V3-Base 出發
- 使用基於規則的獎勵函數
- 僅驗證答案正確性和思考格式
- **不使用**神經網絡獎勵模型(避免 reward hacking)

**訓練模板**:
```
簡單的結構化模板:
1. 首先產生推理過程
2. 然後呈現最終答案
約束僅限於結構格式,避免內容偏見
```

**「啊哈時刻」**:
- 模型自發產生推理行為
- 無需明確編程複雜推理模式
- 通過 RL 自然湧現

**性能**:
- AIME 2024 pass@1 接近 OpenAI o1-0912
- 使用多數投票時超越 OpenAI o1-0912
- 改進是內在的,由延長的測試時計算驅動

**2. DeepSeek-R1: 多階段訓練流程**

雖然 R1-Zero 證明了純 RL 的可行性,但 R1 採用了更實用的多階段方法:

**階段 1: 冷啟動 (Cold Start)**
- 使用專家模型生成推理數據
- 專家模型訓練:
  - 從未公開的 base model 開始
  - SFT on <problem, response> 數據
  - SFT on 合成的 <system prompt, prompt, problem, R1 response> 數據
  - RL 使用未公開的獎勵函數
- 每個專家模型專注於一個特定領域(數學、編程、邏輯)

**階段 2: 推理數據生成**
- 使用內部 DeepSeek-R1-Lite 生成合成數據
- 系統提示要求模型在思考過程中反思和驗證

**階段 3: 監督微調 (SFT)**
- 2 個 epochs
- 1.5M 樣本
- 推理數據(數學、編程、邏輯)+ 非推理數據(創意寫作、角色扮演、簡單問答)
- 非推理數據由 DeepSeek-V2.5 生成並經人工檢查

**階段 4: 強化學習**
- 使用 GRPO (Group Relative Policy Optimization)
- 基於規則的獎勵模型
- 避免神經網絡獎勵模型的 reward hacking 問題

**3. 獎勵工程 (Reward Engineering)**

DeepSeek 開發了一套基於規則的獎勵系統:
- **準確性檢查**: 答案是否正確
- **格式檢查**: 是否遵循思考-回答結構
- **優勢**: 
  - 不會過度擬合獎勵模型
  - 避免 reward hacking
  - 更穩定的訓練過程

**4. 知識蒸餾 (Distillation)**

成功將能力壓縮到小模型:
- **最小模型**: 1.5B 參數
- **方法**: 使用 R1 生成的數據訓練小模型
- **效果**: 小模型保留了大模型的推理能力

#### 湧現行為 (Emergent Behaviors)

R1 訓練過程中自發出現的行為:
- **自我反思**: 檢查自己的推理過程
- **多步驗證**: 用不同方法驗證答案
- **錯誤修正**: 發現錯誤並重新思考
- **複雜推理鏈**: 構建多步邏輯推理

這些行為**並未明確編程**,而是通過 RL 自然湧現。

#### 模型規格與性能

**基礎架構**:
- 基於 DeepSeek-V3
- 671B 總參數
- 37B 激活參數
- 128K 上下文長度

**性能對比**:
- 與 GPT-4o-0513 相當
- 與 Claude-3.5-Sonnet-1022 相當
- 與 OpenAI o1 在推理任務上競爭

**成本優勢**:
- 訓練成本遠低於競品
- 推理成本也更低
- API 定價極具競爭力

#### 市場影響

**2025年1月20日發布後**:
- DeepSeek AI 助手(移動 app)迅速登頂 Apple App Store
- 超越 ChatGPT 成為下載量第一

### 2.5 mHC: 架構層面的根本性創新 (2024年12月)

**論文**: mHC: Manifold-Constrained Hyper-Connections  
**arXiv**: 2512.24880v2  
**發布日期**: 2024年12月(2026年1月更新)

這是本報告重點分析的最新論文,代表了 DeepSeek 在架構創新上的最新突破。

#### 背景: 殘差連接的演進

**傳統殘差連接 (Residual Connection)**

自 ResNet (He et al., 2016) 以來,殘差連接成為深度學習的基石:

```
x_{l+1} = x_l + F(x_l, W_l)
```

**關鍵特性: 恆等映射 (Identity Mapping)**

遞迴展開多層:
```
x_L = x_l + Σ(i=l to L-1) F(x_i, W_i)
```

恆等映射的作用:
- x_l 本身直接傳遞到 x_L,無任何修改
- 確保訊號在每一層都不會被破壞
- 梯度可以直接反向傳播,避免梯度消失
- 訓練穩定性的關鍵

**局限性**:
- 資訊只能走一條路
- 通道寬度有限
- 拓撲複雜度受限

#### Hyper-Connections (HC) 的突破與問題

**HC 的創新 (Zhu et al., 2024)**

擴展殘差流寬度:
```
x_{l+1} = H_l^res x_l + (H_l^post)^T F(H_l^pre x_l, W_l)
```

其中:
- x_l 從 C 維擴展到 n×C 維
- H_l^res ∈ R^{n×n}: 可學習的殘差映射矩陣
- H_l^pre ∈ R^{1×n}: 聚合特徵到層輸入
- H_l^post ∈ R^{1×n}: 將層輸出映射回殘差流

**優勢**:
- 大幅增加拓撲複雜度
- 不增加單個計算單元的 FLOPs
- 實驗顯示性能提升

**嚴重問題**:

1. **破壞恆等映射**

遞迴展開多層 HC:
```
x_L = (∏_{i=1}^{L-l} H_{L-i}^res) x_l + ...
```

問題:
- 複合映射 ∏ H^res 無法保持特徵的全域均值
- 導致無界的訊號放大或衰減
- 訓練不穩定,難以擴展到大規模

2. **記憶體訪問開銷**

- 殘差流寬度擴展 n 倍
- 記憶體訪問成本大幅上升
- 硬體效率未充分考慮

3. **可擴展性受限**

- 深層網絡訓練容易數值爆炸
- 限制了 HC 在大規模模型中的應用

#### mHC: 流形約束的解決方案

**核心思想**

將 HC 的殘差連接空間投影到**特定流形 (Manifold)**上,恢復恆等映射特性,同時保持 HC 的拓撲優勢。

**數學基礎: Birkhoff Polytope**

Birkhoff polytope 是所有**雙隨機矩陣 (Doubly Stochastic Matrices)** 的集合:
- 矩陣的每一行和每一列的和都等於 1
- 所有元素非負

**性質**:
- 雙隨機矩陣的乘積仍是雙隨機矩陣(乘法封閉性)
- 雙隨機矩陣保持向量的均值不變
- 嚴格控制訊號範數,防止爆炸或消失

**投影方法: Sinkhorn-Knopp 算法**

使用 Sinkhorn-Knopp 算法 (1967) 將 H_l^res 熵投影到 Birkhoff polytope:

```
演算法流程:
1. 初始化: M = H_l^res
2. 迭代:
   - 行正規化: M = M / sum(M, axis=1)
   - 列正規化: M = M / sum(M, axis=0)
3. 重複直到收斂
```

結果:
- H_l^res 成為雙隨機矩陣
- H_l^res x_l 是輸入特徵的凸組合
- 特徵均值守恆,訊號範數受控

**恆等映射恢復**

因為雙隨機矩陣的乘法封閉性:
```
∏_{i=1}^{L-l} H_{L-i}^res 仍是雙隨機矩陣
```

結果:
- 深層之間的恆等映射得以維持
- 訓練穩定性大幅提升
- 支持任意深度的模型

#### 效率優化: 基礎設施設計

**1. Kernel Fusion (核心融合)**

使用 TileLang (Wang et al., 2025) 開發混合精度 kernel:
- 將多個操作融合到單個 CUDA kernel
- 減少記憶體訪問次數
- 提高計算吞吐量

**2. Selective Recomputing (選擇性重計算)**

針對記憶體佔用優化:
- 前向傳播時不存儲某些中間激活
- 反向傳播時按需重新計算
- 用計算換記憶體,適合記憶體受限場景

**3. DualPipe 中的通訊重疊**

精心設計的流水線並行:
- 將計算和通訊重疊
- 隱藏通訊延遲
- 最大化 GPU 利用率

#### 實驗結果

**穩定性分析**

在 270 億參數等級的大模型中:
- mHC 訓練過程穩定,無數值爆炸
- 損失曲線平滑,無異常波動
- 支持更深的網絡架構

**性能提升**

與 baseline Transformer 相比:
- 在相同計算預算下性能提升明顯

## 四、關鍵技術深度解析

### 4.1 Multi-Head Latent Attention (MLA)

**問題陳述**

標準 MHA 的 KV cache 問題:
```
KV cache size = 2 × num_layers × num_heads × d_head × seq_len

例如:
- 60 層
- 40 個 head
- d_head = 128
- seq_len = 100K

KV cache ≈ 60 × 40 × 128 × 100K × 2 bytes ≈ 57.6 GB
```

這在長上下文場景下成為瓶頸。

**MLA 解決方案**

核心思想:壓縮 K、V 到低維潛在空間

```
傳統 MHA:
Q_i, K_i, V_i ∈ R^{d_head}  (每個 head)

MLA:
K_c, V_c ∈ R^{d_latent}  (壓縮後)
其中 d_latent << d_head × num_heads

推理時:
K = Up_K(K_c)
V = Up_V(V_c)
```

**技術細節**

1. **低秩聯合壓縮**
   - 使用低秩矩陣分解
   - K、V 共享部分壓縮空間
   - 進一步減少參數

2. **解耦 RoPE**
   - 旋轉位置編碼單獨處理
   - 不增加潛在向量維度
   - 保持位置資訊精度

3. **動態解壓**
   - 只在需要時將潛在向量投影回原始維度
   - 避免存儲完整 K、V

**效果量化**

- 記憶體減少: 93%
- 上下文長度: 4K → 128K (32倍)
- 性能: 與標準 MHA 相當或更好
- 推理速度: 在長上下文下顯著提升

### 4.4 mHC 的數學基礎

**Birkhoff Polytope**

定義:
```
B_n = {P ∈ R^{n×n} | P1 = 1, P^T1 = 1, P ≥ 0}
```

其中:
- P1 = 1: 每行和為 1
- P^T1 = 1: 每列和為 1
- P ≥ 0: 所有元素非負

**性質**:

1. **凸組合**
   ```
   Px 是 x 的凸組合
   即 (Px)_i = Σ_j P_{ij} x_j
   其中 Σ_j P_{ij} = 1
   ```

2. **均值守恆**
   ```
   mean(Px) = mean(x)
   ```

3. **範數控制**
   ```
   ||Px||_2 ≤ ||x||_2  (非擴張性)
   ```

4. **乘法封閉**
   ```
   P, Q ∈ B_n ⟹ PQ ∈ B_n
   ```

**Sinkhorn-Knopp 算法**

目標:將任意矩陣 M 投影到 B_n

算法:
```python
def sinkhorn_knopp(M, max_iter=100, tol=1e-6):
    for _ in range(max_iter):
        # 行正規化
        M = M / M.sum(axis=1, keepdims=True)
        # 列正規化
        M = M / M.sum(axis=0, keepdims=True)
        
        # 檢查收斂
        if check_convergence(M, tol):
            break
    return M
```

**收斂性質**:
- 保證收斂到 B_n
- 收斂速度:線性
- 在實踐中很快(通常 < 20 次迭代)

**在 mHC 中的應用**:

```
前向傳播:
1. 計算 H_l^res
2. P_l = SinkhornKnopp(H_l^res)
3. x_{l+1} = P_l x_l + ...

反向傳播:
1. 通過 Sinkhorn 算法反向傳播梯度
2. 更新 H_l^res
```

結果:
- H_l^res 被約束在 B_n 上
- 訓練穩定性大幅提升
- 恆等映射特性恢復

## 附錄 D: 代碼示例

### mHC Sinkhorn-Knopp 投影

```python
import torch

def sinkhorn_knopp(M, max_iter=100, tol=1e-6):
    """
    將矩陣 M 投影到 Birkhoff polytope
    
    Args:
        M: 輸入矩陣 (n x n)
        max_iter: 最大迭代次數
        tol: 收斂容差
    
    Returns:
        投影後的雙隨機矩陣
    """
    M = torch.abs(M)  # 確保非負
    M = M + 1e-8  # 避免除零
    
    for _ in range(max_iter):
        # 行正規化
        M = M / M.sum(dim=1, keepdim=True)
        
        # 列正規化
        M = M / M.sum(dim=0, keepdim=True)
        
        # 檢查收斂
        row_sum = M.sum(dim=1)
        col_sum = M.sum(dim=0)
        if torch.max(torch.abs(row_sum - 1)) < tol and \
           torch.max(torch.abs(col_sum - 1)) < tol:
            break
    
    return M

# 使用示例
H_res = torch.randn(4, 4)  # 殘差映射矩陣
P = sinkhorn_knopp(H_res)
print(f"行和: {P.sum(dim=1)}")
print(f"列和: {P.sum(dim=0)}")
```

### MLA 簡化實現

```python
import torch
import torch.nn as nn

class MultiHeadLatentAttention(nn.Module):
    def __init__(self, d_model, num_heads, d_latent):
        super().__init__()
        self.num_heads = num_heads
        self.d_head = d_model // num_heads
        self.d_latent = d_latent
        
        # 壓縮投影
        self.q_down = nn.Linear(d_model, d_latent)
        self.k_down = nn.Linear(d_model, d_latent)
        self.v_down = nn.Linear(d_model, d_latent)
        
        # 解壓投影
        self.q_up = nn.Linear(d_latent, d_model)
        self.k_up = nn.Linear(d_latent, d_model)
        self.v_up = nn.Linear(d_latent, d_model)
        
        self.out = nn.Linear(d_model, d_model)
    
    def forward(self, x, kv_cache=None):
        batch, seq_len, d_model = x.shape
        
        # 壓縮到潛在空間
        q_latent = self.q_down(x)
        k_latent = self.k_down(x)
        v_latent = self.v_down(x)
        
        # 如果有 cache,使用壓縮的 cache
        if kv_cache is not None:
            k_latent = torch.cat([kv_cache['k'], k_latent], dim=1)
            v_latent = torch.cat([kv_cache['v'], v_latent], dim=1)
        
        # 解壓到原始空間
        q = self.q_up(q_latent).view(batch, seq_len, self.num_heads, self.d_head)
        k = self.k_up(k_latent).view(batch, -1, self.num_heads, self.d_head)
        v = self.v_up(v_latent).view(batch, -1, self.num_heads, self.d_head)
        
        # 標準注意力計算
        q = q.transpose(1, 2)  # (batch, heads, seq_len, d_head)
        k = k.transpose(1, 2)
        v = v.transpose(1, 2)
        
        scores = torch.matmul(q, k.transpose(-2, -1)) / (self.d_head ** 0.5)
        attn = torch.softmax(scores, dim=-1)
        out = torch.matmul(attn, v)
        
        out = out.transpose(1, 2).contiguous().view(batch, seq_len, d_model)
        out = self.out(out)
        
        # 返回壓縮的 KV cache
        new_cache = {'k': k_latent, 'v': v_latent}
        return out, new_cache

# KV cache 大小對比
d_model = 4096
num_heads = 32
d_latent = 512
seq_len = 100000

# 標準 MHA
kv_cache_mha = 2 * num_heads * (d_model // num_heads) * seq_len * 2  # 2 bytes per element
print(f"MHA KV cache: {kv_cache_mha / 1e9:.2f} GB")

# MLA
kv_cache_mla = 2 * d_latent * seq_len * 2
print(f"MLA KV cache: {kv_cache_mla / 1e9:.2f} GB")
print(f"壓縮率: {(1 - kv_cache_mla / kv_cache_mha) * 100:.1f}%")
```
