過去幾年，業界解決 agent「健忘」的方式幾乎都繞著同一個思路：把記憶的容器做大。更大的 context window、外接的向量資料庫、一層又一層的摘要緩衝區。Stanford 團隊在今年七月發表的 AutoMem，提出了一個完全不同的問題——如果「記得好」根本不是容器大小的問題，而是一種模型必須練習、犯錯、再進步的技能，就像人不是靠更大的筆記本學會做筆記，而是靠學會判斷什麼值得寫下來？

這個重新框架的結果，值得認真看待。

## 兩個迴圈，一套架構

AutoMem 的核心，是把記憶管理拆成兩個可以分開優化的迴圈。

<img src="/assets/img/charts/automem_architecture.svg" alt="AutoMem 兩迴圈架構圖：agent trajectories 進入 structure optimization loop 反覆修訂 memory scaffold，再進入 proficiency training loop，最終部署為 frozen task model 與 trained memory specialist">

第一個迴圈是**結構優化**：一個強模型會回顧 agent 完整的行動軌跡，逐步修訂決定記憶如何組織的 scaffold——什麼該寫、怎麼索引、何時取回。這個過程完全不動模型權重，只改變記憶的組織規則，在 Crafter 環境裡經過 v0 到 v5 共六個版本的迭代才收斂，MiniHack 需要五版，NetHack 只需要三版，某種程度反映了任務本身的複雜度。

第二個迴圈才是**能力訓練**：在 scaffold 固定之後，另外訓練模型操作這套 scaffold 的熟練度，也就是論文所說的「+train」階段。這一步疊加在結構優化之上，帶來額外的效能提升。

架構上最值得注意的一點，是任務模型（frozen task model）和記憶專家（memory specialist）被明確拆開——base model 的權重全程不動，只有負責記憶的部分被訓練。這也是為什麼一個開源的 32B 模型（Qwen2.5-32B-Instruct），單靠記憶行為的訓練，就能被拉到與前沿級系統相近的表現區間，完全沒有動用更大的模型或更長的 context window。

論文也把自己放進了 2025 到 2026 這波記憶研究的脈絡裡做定位：MemAct 把記憶操作視為 context window 上的動作；MemSkill 同樣把記憶操作框架成一組可演化的技能，從失敗中檢討成長；MeMo 訓練一個獨立的記憶模型搭配凍結的 base model，架構上與 AutoMem 的拆分推理相近，但鎖定的是靜態文件問答而非長時程的 agent 決策；MemEvolve、EvolveMem 則透過類似的診斷—修訂迴圈演化記憶架構，但僅止於推理階段，從未真正訓練模型在記憶上的熟練度。AutoMem 的區隔在於同時涵蓋結構與熟練度兩個軸線，而不是只優化其中一個。

## 學會寫記憶，不等於學會守護記憶

這裡是這篇文章真正想談的地方。AutoMem 訓練的是「該寫什麼、怎麼組織」，但整套訓練目標裡完全沒有涉及「這筆記憶屬於誰、誰可以讀」的問題。一旦記憶寫入策略本身是學出來的，而不是人工架構的，這個治理缺口就從理論問題變成實際風險——一個被學出來的寫入策略，如果遭到汙染，汙染的不是單一筆記憶，而是寫入這件事本身的判斷力。

這正是 2026 年多租戶、雲端 agentic 系統安全研究關注的核心。

### AgentLeak：外顯輸出乾淨，不代表系統安全

AgentLeak 這個 benchmark 提出了一個關鍵洞察：多數隱私稽核只看最終輸出，但 multi-agent 系統真正的洩漏管道，發生在協調過程的內部通道裡。這套 benchmark 涵蓋醫療、金融、法律、企業四個領域共一千個情境，把七種通道都納入儀器化追蹤——最終輸出、agent 之間的訊息、工具的輸入輸出、共享記憶、日誌、產出物。

實測結果很有意思，也有點違反直覺：多 agent 架構其實**降低**了最終輸出層的洩漏率（27.2% 對比單一 agent 的 43.2%），但因為引入了未受監控的內部通道，把系統整體暴露率推高到 68.9%。agent 之間的訊息通道洩漏率高達 68.8%，遠高於輸出層的 27.2%，這代表只看輸出層的稽核，會漏掉四成以上的違規事件。五個受測模型裡，Claude 3.5 Sonnet 在外部與內部通道的洩漏率都是最低的，暗示模型層級的安全對齊，某種程度上能轉移到內部通道的保護——這對用未經過同等對齊調校的本地端小模型跑多節點架構的人來說，是個值得放在心上的訊號。

### Mnemonic Sovereignty：把記憶治理當成獨立的安全命題

另一篇綜述性論文提出了「記憶主權」（mnemonic sovereignty）這個概念，把持久、可寫入的長期記憶當成獨立的安全客體來分析，而不只是記憶模組底下的技術細節。它借用認知神經科學的框架，把人類記憶的脆弱性對應到 agent 記憶的安全需求：來源監控失敗對應到來源驗證缺失、記憶再固化對應到讀取時被悄悄改寫、社會傳染對應到跨 agent 的記憶擴散。

這篇論文把記憶生命週期拆成六個階段——寫入、儲存、取回、執行、共享、遺忘/回滾——並與四個安全目標交叉檢視：完整性、機密性、可用性、治理。交叉分析後最值得注意的發現是，現有文獻幾乎都集中在寫入與取回時的完整性攻擊，機密性、可用性、以及儲存/遺忘這幾個階段幾乎沒有被認真研究過；而且，目前沒有任何一套已發表的架構，同時涵蓋他們歸納出的九項治理原則。換句話說，這不是有現成標準答案可以照抄的領域，即使是資源充足的團隊，這裡仍然是一片空白。

### 兩者疊加的意義

把這兩篇放在一起看，對 AutoMem 這類「學出來的記憶寫入策略」有個直接的警示：如果記憶寫入者本身是持續在線演化的，一次性的汙染事件就不再是單一筆壞記憶，而會變成永久性的結構性缺陷——因為汙染影響的是判斷力本身，而判斷力會持續影響往後所有的寫入行為。這正是「記憶即技能」這個框架最有意思也最危險的地方：技能學得越好，一旦技能本身出問題，代價也越大。

## 寫在最後

AutoMem 證明了記憶管理可以是一個獨立訓練的技能維度，這個方向的效能提升是實打實的。但論文的訓練目標裡，沒有任何一項獎勵訊號涉及來源驗證、範圍隔離、或治理層級的約束——這代表任何想把類似架構部署到多使用者、多租戶場景的人，都得自己補上這一塊，目前的文獻裡找不到現成的答案。對本地端、單一使用者的架構來說，這個風險目前還只是理論；但只要往雲端多租戶的方向走一步，這就會變成第一個必須解決的問題，而不是後續才補的安全加固。
