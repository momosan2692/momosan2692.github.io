# Agentic Trading System — System Design Document

> **版本：** v0.1 Draft  
> **市場：** 美股（US Equities）  
> **目標用戶規模：** 限定 10 人以內（Private / Closed Beta）

---

## 1. 系統定位與背景

### 1.1 從量化交易到 Agentic Trading

本系統由量化交易（Quantitative Trading）領域出發，核心理念是將傳統演算法交易的嚴謹邏輯，結合現代 LLM Agent 的自然語言推理能力，打造一個**可授權、可審計、可擴展**的智能交易代理平台。

傳統量化交易的限制：
- 策略規則必須由人工完全定義，無法動態理解市場事件（如財報、Fed 聲明）
- 難以整合非結構化資訊（新聞、公告、研究報告）
- 策略調整需要重新編程，缺乏彈性

本系統的突破點：
- 用 **LLM Agent** 理解市場語意，補充傳統演算法的感知盲區
- 用 **RAG（Retrieval-Augmented Generation）** 即時注入最新市場知識
- 用 **CoT（Chain-of-Thought）** 讓每一個交易決策都可追溯、可解釋
- 在用戶授權範圍內，Agent 可作為**交易代理人**自主執行

## 2. 核心技術選擇與理由

### 2.1 為什麼選擇小模型 + RAG + CoT，而非直接使用大模型？

| 維度 | 大模型（GPT-4 等） | 小模型 + RAG + CoT |
|------|------|------|
| 知識時效性 | 有截止日期，不知道最新財報 | RAG 即時注入，永遠最新 |
| 領域深度 | 廣而淺，專業細節容易幻覺 | 領域微調後精準度高 |
| 策略隱私 | 策略傳送至第三方伺服器 | 本地部署，策略不外洩 |
| 推理延遲 | API 延遲不穩定 | 本地推理，延遲可控 |
| 成本 | 長期使用成本極高 | 一次性硬體投資，邊際成本近零 |
| 可解釋性 | 黑盒輸出 | CoT 保留完整推理鏈，可審計 |

**結論：** 在財務金融特定領域，小模型 + 高品質 RAG + CoT 的組合，在實際表現上可以達到甚至超越大模型，同時具備成本、隱私、可控性的全面優勢。

### 2.2 為什麼需要 CoT（Chain-of-Thought）？

LLM 的每次前向傳播（Forward Pass）計算深度是固定的——不管問題多複雜，模型只過一次固定層數的網路。CoT 的核心價值是**用輸出長度換取計算深度**：每個中間推理步驟都是新的 token，等於讓模型在更多計算步驟上處理同一問題。

在交易場景中，CoT 強制 Agent 依照分析框架推理：

```
市場狀態確認 → 訊號驗證 → 策略條件比對 → 風險評估 → 交易意圖生成
```

這不只是提升正確性，更是**合規與審計的必要條件**——每一筆交易都有可追溯的決策日誌。

### 2.3 為什麼需要 Agent + Memory（mem0）？

Context Window 是 LLM 的根本限制。一個活躍交易日的資料量遠超任何模型的上下文視窗：

```
10ms tick data   → 數百萬筆
新聞事件         → 數百條
歷史交易紀錄     → 數千筆
策略文件         → 數萬 tokens
```

**Agent** 的職責是智能地決定「什麼資料、在什麼時機、放入 context」。  
**mem0** 則提供跨對話的持久記憶，讓 Agent 累積對每個用戶的深度理解。

Memory 分為四層：

| 層級 | 內容 | 壽命 |
|------|------|------|
| In-Context Memory | 當前市場狀態、當前對話 | 單次執行週期 |
| Episodic Memory | 交易決策紀錄、CoT 推理鏈 | 長期，供審計 |
| Semantic Memory | 用戶策略偏好、風險承受度 | 持久，持續更新 |
| Procedural Memory | Agent 操作流程、工具呼叫定義 | 系統設計時固化 |

## 3. 資料基礎設施

### 3.1 Datafeeder — 核心競爭優勢

本系統已擁有延遲低於 **1/100 秒（10ms）** 的市場資料饋送（Datafeeder），這是整個系統的護城河。

關鍵設計原則：**10ms 的資料不直接進入 LLM，而是先經過訊號處理層。**

```
原始 tick data（10ms）
        ↓
訊號處理層（純 Algorithm，毫秒級）
  - 技術指標計算（RSI、MACD、VWAP）
  - 異常偵測（成交量爆量、價格跳空）
  - 策略條件掃描
        ↓
事件觸發層（有訊號才喚醒 Agent）
        ↓
LLM Agent（秒級決策）
```

這個設計解決了 LLM 推理時間（500ms～3s）與高頻資料（10ms）之間的天然矛盾——速度敏感的計算不走 LLM，LLM 只在需要語意判斷時才啟動。

### 3.2 RAG 資料庫分層設計

| 層級 | 內容 | 更新頻率 | 共享方式 |
|------|------|----------|----------|
| 靜態知識層 | 財務理論、會計準則、法規 | 低 | 所有用戶共享 |
| 週期更新層 | 財報、年報、法說會逐字稿 | 季度 | 所有用戶共享 |
| 即時層 | 新聞、SEC 公告、Fed 聲明 | 即時 | 所有用戶共享 |
| 個人層 | 用戶策略文件、交易歷史 | 即時 | 嚴格 user_id 隔離 |

**資料格式處理原則：** 進入 LLM 前，結構化數字必須先轉換為有意義的狀態描述：

```
❌ 原始：AAPL price: 189.34, volume: 2,847,293
✅ 轉換：AAPL 過去 15 分鐘成交量是 20 日均量的 2.3 倍，
         價格突破前高 189.20，RSI(14) = 67，接近超買區間
```

## 4. 系統架構

### 4.1 三層交易能力模型

```
Level 1：Algorithm Trading
  人定規則，機器執行，速度優先
        ↓ 升級
Level 2：LLM-Assisted Trading
  AI 分析訊號，人做最終決策
        ↓ 升級
Level 3：Agentic Trading
  AI Agent 在授權範圍內自主執行交易
```

用戶可以選擇停在任意層級，三層不是替代關係而是遞進關係。

### 4.2 系統架構圖

```
┌─────────────────────────────────────┐
│           API Gateway               │
│    JWT 身份驗證 + Rate Limiting      │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│        Agent Orchestrator           │
│   任務分派 / 優先級管理 / 工具呼叫   │
└──┬───────────────┬───────────────── ┘
   ↓               ↓
┌──────────┐  ┌────────────────────────┐
│ Signal   │  │    Memory Router       │
│ Engine   │  │  mem0 + pgvector       │
│(10ms 層) │  │  user_id namespace 隔離│
└──┬───────┘  └────────────────────────┘
   ↓
┌─────────────────────────────────────┐
│      vLLM Inference Server          │
│   Qwen2.5-14B-AWQ / RTX 4090       │
│   prefix cache 共享市場狀態描述     │
└──────────────┬──────────────────────┘
               ↓
┌──────────────────────────────────────┐
│  Shared RAG         │  Private RAG   │
│  市場知識共用向量庫  │  個人策略/偏好 │
└──────────────────────────────────────┘
               ↓
┌─────────────────────────────────────┐
│         Guardrail Layer             │
│   權限檢查 / 風控熔斷 / 合規過濾    │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│        Broker API                   │
│   Alpaca / Interactive Brokers      │
└─────────────────────────────────────┘
```

### 4.3 Guardrail — 授權與風控設計

這是 Agentic Trading 的信任核心，用戶自行設定 Agent 的行為邊界：

```
可交易的商品清單          → 例：僅限 S&P 500 成分股
單筆最大金額              → 例：$5,000
每日最大損失上限（熔斷）  → 例：總資產 2%
持倉時間限制              → 例：不持有隔夜部位
人工確認門檻              → 例：超過 $10,000 需推播確認
禁止交易時段              → 例：重大經濟數據發布前 5 分鐘
```

Agent 在此沙盒內自主運作，超出任何一條邊界立即停止並通知用戶。

## 5. 為什麼限定在 10 人以內？

這是本系統最重要的設計決策，理由是多維度的。

### 5.1 硬體資源的精準匹配

```
10 人並發 × Qwen2.5-14B-AWQ
→ 單張 RTX 4090（24GB VRAM）完全足夠
→ 模型佔用約 10GB，剩餘處理 KV Cache 和並發請求
→ 一次性硬體成本約 $1,500
→ vs 雲端 GPU 長期租用：$2-3/小時，年成本逾 $15,000+
```

10 人的規模使得本地部署在**成本效益**上遠優於雲端方案，未來若擴展至 50 人，僅需增加一張卡，架構無需重構。

### 5.2 深度個人化成為可能

10 人規模讓每個用戶都能獲得真正的深度個人化，而非「選擇策略模板」：

- Agent 真正學習每個用戶的交易風格（激進 vs 保守）
- 記憶哪類訊號對這個用戶歷史上有效
- 理解用戶對特定板塊的偏好
- 調整推理框架以符合個人風險承受度

**這種程度的個人化在大規模系統中幾乎不可能實現。**

### 5.3 記憶品質的人工監管

10 人規模讓創始人可以**親自審查每個用戶的 Agent 記憶品質**，確保 mem0 學到的是有效的策略偏好，而非市場噪訊。這在早期產品迭代中是不可替代的競爭優勢。

### 5.4 信任關係本身就是產品

10 個用戶意味著可能認識每一個人，授權設定可以一對一討論，交易策略可以深度客製化。這種**高信任、高接觸**的服務模式是早期量化工具產品的核心差異化。

### 5.5 合規風險的可控性

Agentic Trading 在美國證券法規下需要謹慎定位：

- **工具定位（合規）：** 用戶設定規則，Agent 執行 → 屬於自動化工具，無需 RIA 執照
- **顧問定位（需監管）：** AI 主動推薦投資策略 → 可能涉及 Registered Investment Advisor 規定

10 人規模讓每個用戶的授權設定、策略定義、操作記錄都可以清晰維護，合規舉證能力遠優於大規模平台。

### 5.6 資訊共享策略的彈性

| 共享層 | 內容 | 設計 |
|--------|------|------|
| 共享 | 市場新聞、財報解讀、SEC 公告 | 所有用戶共用 RAG，越用越好 |
| 不共享 | 個人策略、交易紀錄、持倉資訊 | 嚴格 user_id namespace 隔離 |
| 可選擇 | 策略訊號廣播 | 明確 opt-in，預設關閉 |

10 人的封閉環境讓「市場知識共享、個人策略隔離」的混合設計更容易維護信任。

## 6. 技術選型總覽

| 元件 | 選型 | 理由 |
|------|------|------|
| LLM 推理框架 | vLLM | 最佳多用戶並發效能，支援 prefix cache |
| 基礎模型 | Qwen2.5-14B-AWQ | 14B 甜蜜點，量化後單卡可運行 |
| Agent 框架 | LangGraph 或自建 | 支援複雜工作流，狀態管理清晰 |
| 記憶系統 | mem0 + pgvector | 持久語意記憶 + 向量搜尋 |
| RAG 向量庫 | Qdrant / pgvector | 支援 user_id namespace 過濾 |
| Broker API | Alpaca / IBKR | 美股支援完整，REST API 成熟 |
| 硬體 | RTX 4090 × 1 | 10 人並發足夠，性價比最高 |

## 7. 開發路徑

```
Phase 1：單用戶 MVP
  - 完整交易決策流程跑通
  - Datafeeder → Signal Engine → Agent → Broker API
  - CoT 推理鏈存檔

Phase 2：記憶層整合
  - mem0 接入，Agent 開始累積用戶偏好
  - Episodic Memory 建立（決策審計日誌）

Phase 3：多用戶隔離驗證
  - user_id namespace 隔離正確性驗證
  - Guardrail 壓力測試

Phase 4：共享層優化
  - 市場 RAG 提取為共用層
  - vLLM prefix cache 效益驗證

Phase 5：Agentic Trading 上線
  - 授權框架完整測試
  - 合規文件建立
  - 10 人封閉測試
```

## 8. 核心競爭優勢總結

```
量化背景    → 策略邏輯設計能力，懂得定義有效的 Trigger 條件
10ms Datafeeder → 訊號品質優於大多數零售平台，難以複製
本地 LLM   → 策略隱私保護，零邊際推理成本
CoT 審計鏈 → 每筆交易可解釋，建立用戶信任
10 人限制  → 深度個人化 + 高信任服務模式
```

**最難被複製的部分：懂策略的人設計 AI 系統，而非 AI 工程師猜金融邏輯。**
