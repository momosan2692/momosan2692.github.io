## 1. 引言：自回歸生成的效率困境

大型語言模型（LLM）雖然具有強大的文字生成能力，但其底層架構——**自回歸生成（Autoregressive Generation）**——天生帶有效率瓶頸。

每生成一個新 token，模型必須重新處理**從頭到當前位置的完整序列**，計算所有先前 token 的 Q、K、V 投影矩陣。這意味著：

- 第 1 個 token：處理 1 個 token
- 第 2 個 token：重新處理 2 個 token（其中 1 個與前次完全相同）
- 第 N 個 token：重新處理 N 個 token（前 N-1 個完全相同）

這種重複計算使注意力機制的複雜度隨序列長度呈 **二次方增長（O(n²)）**，是生成長文本時速度急劇下降的根本原因。

以一個實驗為例：啟用 KV 快取時，生成 1,000 個 token 約需 21.7 秒；關閉快取則需約 107 秒，**相差約 5 倍**。

## 2. KV 快取的本質

> KV 快取是 GPU 高速記憶體（VRAM）中的一塊專用空間，用於儲存 Transformer 自注意力層已計算過的 **Key（K）** 和 **Value（V）** 向量，避免在生成每個新 token 時重複執行相同的矩陣乘法。

![kv-cache-concept-diagram](/assets/img/header/2026-04-16/kv-cache-concept-diagram.png)

*圖示：KV 快取概念圖——先前 token 的 K/V 向量被儲存並重複使用，僅對新 token 計算 Query。*

KV 快取之所以有效，基於三個數學事實：

1. **先前 token 不會改變**：它們的 Embedding 與 K/V 向量在生成過程中保持恆定。
2. **只有 Query 需要更新**：每個新 token 只需計算自己的 Q 向量，再與快取中所有歷史 K 向量做內積（`QKᵀ / √d`）。
3. **注意力依賴位置**：新 token 透過 Query 對所有先前 token 的 Key 求注意力分數，再對 Value 加權求和，即可完成一步解碼。

從 O(n²) 降至 **O(n)**，這是 KV 快取的核心貢獻。

## 3. KV 快取如何運作：逐步解析

以輸入 "Time flies fast" 為例：

**Step 1 — Prefill（預填充）**

輸入序列中所有 token（"Time", "flies"）一次性並行通過模型，計算每層的 K、V 張量並儲存至快取。

```
Cache state after prefill:
Layer 0: K=[K_time, K_flies], V=[V_time, V_flies]
Layer 1: K=[K_time, K_flies], V=[V_time, V_flies]
... × L layers
```

**Step 2 — Decode（解碼循環）**

生成 "fast" 時：
1. 僅將 "flies"（最後一個 token）的 Embedding 送入模型
2. 計算新 token 的 Q 向量：`Q_fast`
3. 從快取讀取所有歷史 K：`[K_time, K_flies]`
4. 計算注意力分數：`softmax(Q_fast · [K_time, K_flies]ᵀ / √d)`
5. 以分數加權 `[V_time, V_flies]` 得到輸出
6. 將新的 `K_fast, V_fast` 附加進快取

如此循環，每步只做一個 token 的前向傳播，而不是重新處理整段歷史。

## 4. KV 快取大小計算公式

$$\text{KV Cache Size} = 2 \times L \times H_{kv} \times D \times S \times B \times P$$

| 符號 | 含義 | 說明 |
|------|------|------|
| $2$ | K + V | 兩組矩陣各一份 |
| $L$ | 層數（Layers） | 模型 Transformer Block 總數 |
| $H_{kv}$ | KV 注意力頭數 | 若使用 GQA/MQA，為縮減後的 KV head 數，**非** Query head 數 |
| $D$ | 頭維度（Head Dim） | 每個注意力頭的向量維度 |
| $S$ | 序列長度（Sequence Length） | 上下文 token 數 |
| $B$ | 批次大小（Batch Size） | 並行請求數 |
| $P$ | 精度位元組（Precision） | FP16 = 2 bytes；INT8 = 1 byte；FP4 ≈ 0.5 bytes |

**關鍵等式**：$H_{kv} \times D = d_{kv}$，即鍵值的隱藏維度。在 GQA 模型中，$H_{kv}$ 已是縮減後的值（例如 Llama 3 70B 的 GQA 設定下，$H_{kv}=8$，遠小於 Query head 數 64）。

## 5. 常見模型快取大小範例（含驗算修正）

### Llama 3 70B 架構參數

| 參數 | 數值 |
|------|------|
| 層數 $L$ | 80 |
| KV head 數 $H_{kv}$（GQA） | 8 |
| Head Dim $D$ | 128 |
| 精度 $P$ | FP16 = 2 bytes |

**每個 token 每個批次的快取大小：**

$$\text{Per token} = 2 \times 80 \times 8 \times 128 \times 2 = 327{,}680 \text{ bytes} \approx 320 \text{ KB/token}$$

**範例試算：**

| 配置 | 計算 | 快取大小 |
|------|------|----------|
| S=128k, B=1 | 320 KB × 131,072 | **≈ 39.3 GB** |
| S=2048, B=512 | 320 KB × 2,048 × 512 | **≈ 320 GB** ⚠️ |

> **修正說明**：部分文獻引用 S=2048, B=512 的快取為「~512 GB」，但以 Llama 3 70B 實際 GQA 架構代入計算，正確值為約 **320 GB**。即便如此，「遠超 130 GB 模型權重本身」的結論仍然成立。

## 6. 快取實作策略：DynamicCache vs. StaticCache

![dynamic-vs-static-cache-diagram](/assets/img/header/2026-04-16/dynamic-vs-static-cache-diagram.webp)

*圖示：DynamicCache（動態增長）vs. StaticCache（預分配固定大小）記憶體行為對比。*

Hugging Face Transformers 等框架提供兩種主要快取實作：

### DynamicCache（動態快取）

- **機制**：快取張量隨 token 生成逐步追加（append），形狀不斷改變。
- **優點**：記憶體使用靈活，不會過度預分配。
- **缺點**：動態形狀改變使 JIT 編譯器（如 `torch.compile`）無法充分優化計算圖，限制推理速度上限。

### StaticCache（靜態快取）

- **機制**：在推理開始前，依照**最大序列長度**一次性預分配固定大小的零張量，推理時原地填入。
- **優點**：靜態形狀讓 `torch.compile` 能完整優化生成循環，帶來顯著加速。
- **缺點**：即使實際序列很短，也會佔用最大長度的記憶體，可能造成浪費。

**選擇原則**：
- 需要最高速度、硬體充足 → StaticCache
- 需要靈活處理不同長度的請求 → DynamicCache

## 7. 超出 VRAM 的解決方案：快取卸載

當 KV 快取過大無法完全放入 GPU VRAM 時，業界採用**分層儲存（Tiered Storage）**策略：

![kv-cache-tiered-storage-diagram](/assets/img/header/2026-04-16/kv-cache-offloading-png.webp)
*圖示：KV 快取分層儲存架構——VRAM → CPU RAM → NVMe SSD → 遠端物件儲存（S3）。*

| 儲存層級 | 延遲 | 適用場景 |
|----------|------|----------|
| GPU VRAM（HBM） | ~微秒 | 活躍 session、當前生成中的 token |
| CPU RAM | +10–50 ms | 近期 session、prefix cache 暫存 |
| NVMe SSD | +100–500 ms | 長上下文歷史、不活躍 session |
| 遠端儲存（S3 等） | +秒級 | 持久化 prefix cache、跨節點共享 |

**LMCache** 是此領域的代表性開源工具，支援在 GPU、CPU、本地磁碟與遠端儲存之間透明地移動 KV 快取，並提供：
- **prefill-once, reuse-everywhere** 語意：任何 session 中計算過的 KV 只需 prefill 一次，後續請求可直接載入。
- 與 vLLM / SGLang 深度整合，搭配 NVIDIA Dynamo 可達 3–10 倍的延遲降低。

## 8. 快取剔除策略

當快取達到容量上限時，必須決定「遺忘」哪些 token 的 KV，即**快取剔除（Cache Eviction）**。

![cache-eviction-strategies-diagram](/assets/img/header/2026-04-16/YOUR_SVG_LINK_HERE)
*圖示：不同剔除策略對應的 token 保留模式——Full Cache（完整）、Sliding Window（滑動視窗）、H2O（注意力分數導向）、StreamingLLM（注意力沉降點保留）。*

### 主要策略

**Sliding Window（滑動視窗）**

僅保留最近 $W$ 個 token 的 KV 快取，舊 token 自動被捨棄。

- 代表：Mistral 7B（window size = 4096）
- 記憶體上限：約為 $W$ 個 token 的 KV，不再線性增長
- 缺點：對需要長程依賴的任務有損失

**H2O（Heavy Hitter Oracle）**

依據歷史注意力分數，保留獲得最多注意力的「重要 token」（Heavy Hitter），剔除得分最低的 token。

- 注意力分數越高 = 該 token 越常被其他 token 關注 = 越重要
- 缺點：不適合多輪對話（被剔除的 token 在未來輪次可能再度變得重要）

**StreamingLLM / Attention Sink**

保留最初的少量「注意力沉降點（Attention Sink）」token（通常是前 4–8 個）加上最近的滑動視窗 token，中間的 token 被剔除。

- 啟示：LLM 普遍對序列開頭 token 有極高的注意力分數，這些 token 必須保留，否則整個注意力機制會失穩。
- 優點：可支援理論上無限長度的生成，記憶體固定

**RadixAttention（SGLang 實作）**

將 KV 快取組織成前綴樹（Radix Tree），配合 LRU 剔除策略，讓擁有共同前綴的多個請求**共享**相同的快取塊，大幅減少重複儲存。

## 9. LLM 推理各階層記憶體詳解

LLM 推理期間的記憶體可依**生命週期**（靜態→動態→瞬態）分為五層：

![llm-inference-memory-layers-diagram](/assets/img/header/2026-04-16/YOUR_SVG_LINK_HERE)
*圖示：LLM 推理記憶體階層——從常駐的模型權重，到瞬態的 Activation Buffer，生命週期由長至短排列。*

### 第一層：模型權重（Model Weights）〔靜態〕

- **內容**：所有 Transformer 層的 Q/K/V/O 投影矩陣、FFN 上下行投影、Embedding Table、LayerNorm 參數
- **生命週期**：模型載入後常駐 VRAM，整個服務期間不釋放
- **大小規則**：$\approx 2 \times \text{參數量（bytes，FP16）}$
  - Llama 3 70B：約 130 GB（FP16）
- **特性**：純讀取，可透過 Tensor Parallelism 切片至多張 GPU

### 第二層：KV 快取（KV Cache）〔動態〕

- **內容**：每個 Transformer 層、每個 session、每個 token 的 K/V 張量
- **生命週期**：session 期間持續存在；session 結束或被 evict 時釋放
- **大小規則**：見第 4 節公式，隨序列長度線性增長
- **特性**：長上下文或高並發時是最大動態記憶體消耗源，可能超過模型權重

### 第三層：Activation Buffers（激活緩衝）〔瞬態〕

- **內容**：前向傳播過程的中間張量——注意力分數 $QK^T/\sqrt{d}$、Softmax 輸出、FFN 中間層激活值
- **生命週期**：每次 forward pass 後即可釋放，屬於瞬態記憶體
- **大小規則**：$\approx B \times S \times d_{model}$（遠小於 KV 快取）
- **特性**：本地推理中通常一次只服務一個 session，共用同一份 activation 記憶體

### 第四層：Logit + Sampling Buffer（解碼輸出緩衝）〔極短暫〕

- **內容**：最終輸出的 logit 張量（$B \times \text{vocab\_size}$）以及 top-k/top-p/temperature sampling 所需的臨時緩衝
- **生命週期**：每個 token 解碼步驟分配，採樣後即釋放
- **大小規則**：$\approx B \times |\text{vocab}| \times P$（例如 B=1, vocab=128k, FP32 → 約 512 KB，極小）

### 第五層：CPU RAM / SSD Swap（溢出儲存）〔被動〕

- **內容**：VRAM 承壓時溢出的 KV 快取、被換出的非活躍 session
- **觸發條件**：VRAM 池耗盡，eviction 策略啟動
- **延遲代價**：CPU offload 約 +10–50 ms；SSD swap 約 +100–500 ms

## 10. 分散式推理中的 KV 快取管理

大規模雲端推理系統面臨的核心問題是：**多個 GPU 節點上的 KV 快取分布不均**，導致同一 session 的請求被路由到沒有其快取的節點，強迫重新計算（re-prefill），浪費算力並增加首字延遲（TTFT）。

![distributed-kv-routing-diagram](/assets/img/header/2026-04-16/YOUR_SVG_LINK_HERE)
*圖示：NVIDIA Dynamo 分散式 KV 快取路由——Smart Router 查詢全域 Radix Tree，將請求導向快取命中率最高的 GPU worker。*

### NVIDIA Dynamo Smart Router

**核心資料結構：全域 Radix Tree（前綴樹）**

Dynamo 的 KV 快取管理器維護一個覆蓋整個 GPU 叢集的 Radix Tree，記錄所有快取塊的位置：

- **KVPublisher**：每個 worker 在計算 KV 快取時，將事件發布到全域索引
- **KVIndexer**：接收所有 worker 的事件，維護全域前綴樹與快取位置映射
- **路由成本函數**：`cost = KV cache miss rate + load factor`

當新請求到達時，Smart Router 計算每個 worker 的快取命中率與當前負載，選擇綜合成本最低的 worker，而非簡單地選命中率最高的（高命中 + 高負載 ≠ 最優）。

### LMCache 整合

LMCache 作為 KV 快取管理的可插拔層，整合進 Dynamo 架構，實現：

- **prefill-once, reuse-everywhere**：任何 worker 計算的 KV 快取均可被其他 worker 重複使用
- **多層儲存支援**：GPU → CPU → NVMe → Redis → S3，自動管理提升與降級
- 搭配 WEKA 儲存系統的測試顯示，8 張 H100 的讀取吞吐可達 270 GB/s

## 11. Cloud LLM vs. Local LLM 記憶體配置對比

![cloud-vs-local-memory-diagram](/assets/img/header/2026-04-16/cloud_vs_local_llm_memory.svg)
*圖示：Cloud LLM（多租戶分散式 GPU 叢集）與 Local LLM（單裝置多 session）的 VRAM 配置對比——KV 快取隔離性、容量上限彈性、CPU RAM 用途三大維度的差異。*

### 同一之處

| 記憶體區域 | 特性 | Cloud | Local |
|-----------|------|-------|-------|
| **模型權重** | 常駐 VRAM，唯讀 | 相同（Cloud 為 tensor-parallel 切片） | 相同（單份完整載入） |
| **Activation Buffer** | 瞬態，forward pass 後釋放 | 相同 | 相同（共用單份） |
| **Logit Buffer** | 極短暫，逐 token 分配 | 相同 | 相同 |

### 核心差異

| 維度 | Cloud LLM | Local LLM（多 session） |
|------|-----------|------------------------|
| **KV 快取隔離性** | 每個 session 佔用**獨立** VRAM 區段 | 所有 session 共用**同一**記憶體池 |
| **容量上限** | 軟性——橫向擴展 GPU 節點即可增容 | 剛性——VRAM 滿即觸發 eviction 或 swap |
| **Session 3+ 行為** | 繼續分配獨立區段；路由至新節點 | 觸發 KV eviction，溢出至 CPU RAM/SSD |
| **CPU RAM 用途** | 主動：請求排隊 + Prefix Cache 持久化（降低 TTFT） | 被動：VRAM 壓力下的快取卸載目標 |
| **Activation 排程** | 多 GPU 可並行多個 session 的 forward pass | 單裝置通常序列排程，共用一份 activation |
| **Prefix Cache 共享** | 系統提示可在多個 session 間共享快取塊 | 需框架支援（如 vLLM APC），但池小效果有限 |

## 12. 進階優化技術彙整

### PagedAttention（vLLM）

受作業系統虛擬記憶體分頁機制啟發：

- 將 KV 快取劃分為**固定大小的 KV Block**（類比記憶體頁面）
- 使用**邏輯→物理映射表（Block Table）** 管理非連續儲存
- 支援 Copy-on-Write：多個 sequence 共享相同前綴的 KV block，修改時才複製
- 效果：記憶體碎片率從 60–80% 降至 **< 4%**，吞吐量提升 2–4 倍

### KV 快取量化（KV Cache Quantization）

| 精度降低方式 | 記憶體節省 | 品質影響 |
|------------|-----------|---------|
| FP16 → INT8 | 50% | 微小 |
| FP16 → INT4 | 75% | 小幅 |
| FP16 → NVFP4（Blackwell） | 75% | 需硬體支援 |

### Grouped Query Attention（GQA）

GQA 在**模型訓練階段**即縮減 KV head 數量（$H_{kv} < H_{q}$），讓多個 Query head 共享同一組 KV，從源頭壓縮快取大小：

- MHA：$H_{kv} = H_{q}$（全尺寸快取）
- GQA：$1 < H_{kv} < H_{q}$（縮減快取，保持大部分品質）
- MQA：$H_{kv} = 1$（最小快取，有較明顯品質損失）

Llama 3 70B 採用 GQA（$H_{q}=64, H_{kv}=8$），快取大小為等效 MHA 的 **1/8**。

### Speculative Decoding 中的雙快取管理

使用小型 draft model 先生成候選 token 序列，再由大型 target model 並行驗證：

```python
draft_cache = KVCache(small_model_config)
target_cache = KVCache(large_model_config)
draft_tokens = draft_model.generate(prompt, cache=draft_cache)
verified_tokens = target_model.verify(draft_tokens, cache=target_cache)
```

需同時管理兩套快取，但整體吞吐量大幅提升。

## 13. 總結

KV 快取是 LLM 推理工程中最關鍵的單一優化技術，其重要性體現在：

1. **從 O(n²) 降至 O(n)**：根本解決自回歸生成的計算重複問題
2. **記憶體瓶頸是新主戰場**：在 128k+ 上下文、大批次場景下，快取大小可超越模型權重本身
3. **優化技術形成完整生態**：從實作層（Dynamic/Static Cache）、演算法層（PagedAttention、GQA、量化）到系統層（LMCache、NVIDIA Dynamo），各個維度均有成熟方案

理解 KV 快取，是設計高效 LLM 推理系統的必要基礎。
