> 截斷只解決「不要無限跑下去」，不解決「給我一個有意義的答案」。這兩件事必須用不同的機制分開處理。

---

## 前言：一個關於 Ollama Modelfile 的查證，揭開了更大的問題

事情的起點，是另一個 AI 助手給的一份 Ollama Modelfile 設定建議——教人用 `repetition_penalty`、`presence_penalty` 解決 Qwen3-8B 和 Gemma4 在 Thinking 模式下的「厄運循環」問題。

這份建議看起來很專業，附有完整的步驟、程式碼、甚至承諾「能根除 90% 以上的厄運循環」。

但仔細查證後，發現至少有三個問題：

1. **參數名稱錯誤**：Ollama Modelfile 的 `PARAMETER` 沒有 `repetition_penalty` 和 `presence_penalty` 這兩個名稱，正確的是 `repeat_penalty` 和 `repeat_last_n`。如果照抄這份設定檔去執行，這兩行參數會被靜默忽略。
2. **機制描述過度簡化**：「`<think>` 標籤卡關＝機率分佈卡死」這個說法忽略了一個更精確的可能性——thinking mode 不收斂，往往是 stop token 訓練覆蓋率不足，而不是純粹的重複機率問題。
3. **數字憑空捏造**：「90% 以上」沒有任何來源支撐，是那種「聽起來很科學但其實是杜撰」的典型案例。

這個查證過程，意外地把一個簡單的「調參數」問題，撬開成一整套關於**本地 LLM 推理架構設計**的思考——從單一模型的行為調校，到多節點協調的系統工程，到「該不該用 fine-tune 解決」的邊界判斷。

這篇文章記錄這個思考過程的完整脈絡。

---

## 第一章：先搞清楚事實——Ollama 到底支援什麼參數

在討論任何解法之前，必須先把基礎事實釘死。Ollama 的 Modelfile `PARAMETER` 指令，支援的是模型推理的底層採樣參數，名稱對應的是 llama.cpp 的命名慣例，**不是** OpenAI API 的命名慣例。這是一個常見的混淆來源——很多人習慣 OpenAI API 的 `presence_penalty` / `frequency_penalty` 命名，直接搬到 Ollama Modelfile 上，結果參數沒有生效。

正確可用的核心參數：

```dockerfile
FROM qwen3:8b

PARAMETER temperature 0.7
PARAMETER top_p 0.9
PARAMETER repeat_penalty 1.18
PARAMETER repeat_last_n 256
PARAMETER num_predict 2048

SYSTEM """
你是一個精準、高效的 AI 助手。
請直接回答核心問題，嚴禁在回覆中重複相同的句子、段落或邏輯。
如果你發現自己陷入重複描述，請立刻做出一句總結並結束對話。
"""
```

對照表如下：

| 你以為的參數（OpenAI 命名習慣） | Ollama 實際支援的參數 | 是否存在 |
|---|---|---|
| `repetition_penalty` | `repeat_penalty` | 名稱不同，效果類似 |
| `presence_penalty` | （無對應） | 不存在，會被忽略 |
| `frequency_penalty` | （無直接對應，部分效果由 `repeat_penalty` 涵蓋） | 不存在 |
| `max_tokens` | `num_predict` | 名稱不同，效果類似 |
| （無） | `repeat_last_n` | Ollama 特有，控制重複懲罰回溯的視窗長度 |

這個查證的價值，不只是「糾正一個錯誤」。它揭示了一個更普遍的風險：**AI 生成的技術建議，即使邏輯框架聽起來合理，細節層面的具體參數名稱、數字、API 規格，都需要對照官方文件逐一核實**——尤其是當這些建議將被直接寫進 Modelfile、立刻執行的時候。

## 第二章：兩種「卡住」的本質差異——機率塌陷 vs 動力學陷阱

要正確理解「厄運循環」，必須先分清楚它在不同架構下的數學本質完全不同。這不是學術上的吹毛求疵——機制不同，意味著解法的有效邊界也不同。

### 2.1 Transformer 的復讀機：機率分佈的正回饋塌陷

Qwen3、Gemma4 這類 Transformer 架構的小模型（8B 等級），生成下一個 token 的過程，是一個迭代的條件機率採樣。Softmax 函數把 logit 分數轉換為機率分佈時，存在一個正回饋機制：一旦某個 token 在某個上下文中被選中過一次，它在類似上下文再次出現的機率會被放大——尤其是在參數量有限、邏輯容量不足以支撐長鏈推理的小模型上，這個正回饋特別容易被觸發。

這個現象的根本原因，可以歸結為三點：

**機率陷阱**：小模型預測下一個詞的辨別力較弱，一旦某句話重複出現，該句子的權重會暴增，形成不可逆的機率死循環。

**注意力崩塌**：小模型的 Context 控制力較差，生成長文本時逐漸「忘記」前文脈絡，導致在原地打轉。

**邏輯容量不足**：強迫小模型進行深度推理時，有限的參數無法支撐長邏輯鏈，最終只能反覆重複已知結論。

這三個機制，最終都收斂到同一個數學圖像：在「機率空間」中，系統的軌跡像螺旋一樣，不可逆地收縮到一個**穩定不動點**——對應「只生成這一個 token（或同一段文字）」的固定狀態。一旦進入這個不動點，只有外力（提高 `temperature`、加大 `repeat_penalty`、插入 `stop` 標記）能把它打出來。

### 2.2 Thinking 模式的特殊情況：不只是機率問題

這裡有一個容易被忽略、但你在 vMLX 實際偵錯中遇過的細節：**Thinking 模式下的循環，不完全等同於一般文字生成的復讀機問題**。

當 Qwen3、Gemma4 這類模型啟動 Thinking（思維鏈）模式，它們會在 `<think>` 標籤內進行「自我糾錯」式的內心推理。如果思維卡關，表現出來的是「無限循環」——但這個現象的根因，除了機率塌陷，還疊加了另一個獨立問題：**stop token（如 `</think>`）的訓練覆蓋率不足**。

換句話說，模型不一定是「卡在重複同一句話」，而可能是「不斷生成新的、看似在推進的思考內容，但從未觸發收斂到 `</think>` 的訊號」。這種情況下：

- 單純的 `repeat_penalty` 效果有限——因為內容在表面上沒有逐字重複，repeat_penalty 偵測的是 n-gram 重複，抓不到「語意層面在繞圈子但措辭略有不同」的情況。
- 真正有效的對策，往往是 `num_predict` 設上限，或在 system prompt 裡明確限制 thinking 區塊的長度。

這個區分很重要：**「重複同一句話」和「不斷說新內容但始終不收斂」，是兩種不同的失敗模式，需要不同的偵測和應對手段。**

### 2.3 液態神經網路（LFM）的對照：幾何陷阱而非機率陷阱

放在更大的架構光譜中看，Transformer 的復讀機問題，和液態神經網路（如 Liquid AI 的 LFM）的「極限環」問題，雖然表現都是「重複輸出」，但底層機制截然不同。

Transformer 的不動點，對應「零速度」的靜止狀態——隱藏狀態停止變化，輸出永遠一樣。LFM 的極限環，對應「非零速度」的週期狀態——隱藏狀態持續運動，但被困在一個閉合軌道上，輸出呈現週期性重複而非完全相同。

這個差異，給了我們一個重要的啟示：**「打破循環」這件事，沒有一個放諸四海皆準的解法**。不同架構、不同失敗模式，需要不同的偵測邏輯和不同的干預手段。這也是為什麼那份「90% 根除率」的說法值得懷疑——它把所有「重複」現象當成同一個問題來解，忽略了底層機制的多樣性。

## 第三章：截斷的本質侷限——止血不是治療

這是整個討論中最核心的洞見，值得單獨用一章說清楚。

### 3.1 `num_predict` 在做什麼，不在做什麼

`num_predict`（或任何形式的 max token 截斷）的機制非常簡單：到達 token 上限，**強制停止生成**，不管模型當下是否已經得出結論。

這個機制，回答的是一個運維問題：「如何防止 stream 無限跑下去，耗盡資源、卡住整個系統」。

它完全沒有回答另一個問題：「如何讓模型給出一個有意義的答案」。

這兩個問題，看似相關，實際上是兩個獨立的工程目標：

| 目標 | 對應機制 | 解決了嗎 |
|---|---|---|
| 防止資源耗盡、stream 不掛掉 | `num_predict` 截斷 | ✅ 解決 |
| 得到一個收斂、有意義的回應 | （截斷本身不解決） | ❌ 未解決 |

如果模型卡在極限環或 thinking 死循環裡，截斷只是讓它在某個固定 token 數停下——但它停下的那一刻，**內部狀態仍然是「卡住的」，不是「想通的」**。模型不會因為被截斷，而在內部完成任何推理閉合。它就是被攔腰砍斷，輸出到那裡就結束。

用前面討論的極限環比喻：截斷不會把隱藏狀態從循環軌道上拉出來，它只是在某個時間點按下「暫停鍵」，無論軌跡當時在環上哪個位置。

### 3.2 三種手段的分工表

把這次討論中出現的所有手段做一個完整的分類，可以得到下表：

| 手段 | 作用層面 | 改變了什麼 | 能否產生「結果」 |
|---|---|---|---|
| `num_predict`（截斷） | 運維／資源保護 | 什麼都沒改變，只是停止輸出 | ❌ 不能，純粹止血 |
| `repeat_penalty` / `temperature` | 採樣機率調整 | 改變下一個 token 的選擇機率分佈 | ⚠️ 有機會跳出循環，但不保證 |
| System Prompt 強制結構 | 輸入訊號層 | 改變模型「看到」的上下文，引導它主動收斂 | ✅ 較可能成功，但仍非絕對 |

這張表的核心訊息是：**越靠近「改變輸入訊號」這一端，越有機會讓模型真正產生有意義的結果；越靠近「事後攔截」這一端，越只是在管理損害，而非解決問題。**

### 3.3 如果真的想要「截斷後仍有可用結果」——三個務實做法

**做法一：分段式時間預算 + 強制收尾 prompt**

不要單純設一個硬性 `num_predict` 上限就放著不管，而是設一個較短的軟性 budget（例如 800 token），當生成接近上限時，**主動注入一個新的系統指令**強迫模型立即總結：

```
你已經思考很久了。現在立刻停止推理，
直接給出目前最可能的答案，一句話總結，不要再展開。
```

這個做法的關鍵，是它屬於「改變輸入訊號」這一類手段——透過明確告訴模型「現在該收尾了」，主動引導它的生成軌跡向「給出結論」的方向移動，而不是被動地等它撞到牆。

**做法二：重複偵測 + 局部重新採樣，而非整段截斷**

在 stream 過程中即時偵測是否出現完全重複的 n-gram（例如連續 3 次出現相同句子）。一旦偵測到，**不要截斷整個回應**，而是回退到重複開始前的位置，針對那一段重新採樣（提高 temperature 或加大 `repeat_penalty` 重新生成那一段）。

這個做法的優勢：保留了重複發生前已經產生的有效內容，不會把模型在卡住之前的合理推理也一起丟棄。

**做法三：雙模型備援**

如果你已經有多節點的推理架構（例如本地的 vMLX 多節點佈署），可以讓主模型先嘗試生成；一旦偵測到循環苗頭（重複 n-gram 超過閾值，或 thinking 區塊超過長度仍未閉合），就中止並切換到另一個設定（降低 temperature 重跑，或換用不啟用 thinking mode 的版本直接給答案），而不是讓使用者拿到一段被砍斷的廢話。

這三個做法，都比「單純設一個 `num_predict` 上限就結束」更進一步——它們的共通邏輯，是**把截斷視為觸發點，而不是終點**。截斷之後，系統還需要做一些事，才能真正交付一個有用的結果。

## 第四章：把「截斷後的處理」延伸成系統架構問題

這是整個討論最有意思的轉折點：當截斷後的殘缺輸出，被送進一個多節點架構的 Aggregator，問題就從「怎麼調一個模型的參數」，變成了「怎麼設計一個系統」。

### 4.1 模式：截斷的輸出，變成下一輪的輸入

對於採用多節點推理架構的系統（例如 Aggregator + 多個 specialist 節點的設計），一個自然的想法是：模型卡循環被截斷後，那段殘缺輸出**不應該被直接丟棄**，而是可以被 Aggregator 拿來當作線索，重新構造一個新的 follow-up 問句，再發送回去重試。

流程大致是這樣：

```
specialist 節點生成 thinking → 卡循環 → num_predict 截斷
                    ↓
Aggregator 拿到這段「殘缺輸出」
                    ↓
Aggregator 判斷這段輸出是否有提取價值
                    ↓
若有價值：摘要轉述 + 明確轉向指令，構造新問句
若無價值：直接丟棄，重新發起全新問句
                    ↓
重新發送給 specialist 節點（或換一個 profile）再跑一次
```

這個想法的合理性在於：**截斷前的部分輸出，往往包含了模型卡住前已經摸索到的推理方向**，這個資訊不該被無條件丟棄。

### 4.2 三個關鍵的設計細節

**細節一：判斷殘缺輸出是否值得提取**

截斷的輸出可能是兩種情況：卡循環前已經產生了有意義的部分推理（值得提取重用），或者從一開始就在繞圈子、整段都是噪音（直接丟棄，不要污染下一輪 prompt）。

一個簡單的啟發式：檢查截斷文本裡的重複 n-gram 佔比。如果重複佔比過高（例如超過 40% 的句子是重複的），代表這段輸出本身沒有提取價值，Aggregator 應該重新發起一個全新的問句，而不是把這段噪音當作線索餵回去——否則可能把循環模式本身也一起帶進下一輪 prompt，反而誘發新一輪的循環。

**細節二：新問句的構造方式，決定了是否真的能「轉彎」**

如果 Aggregator 只是把截斷文本整段塞進新 prompt（「之前想到這裡：[殘缺文本]，請繼續」），很可能讓下游模型直接從卡住的地方接著卡——因為輸入訊號幾乎沒變，動力系統還是會被拉回同一個吸引子。

更有效的做法，是 Aggregator 對殘缺輸出做**摘要轉述**而非原文保留，再追加明確的轉向指令：

```
之前的推理方向是：[一句話摘要，去除重複部分]
這個方向似乎沒有收斂。請換一個完全不同的切入角度，
直接給出結論，不要重複之前的論述方式。
```

這等同於主動改變了輸入訊號的性質，比直接把卡住的原始輸出餵回去，更可能讓下游模型跳出同一個循環陷阱。

**細節三：設定重試上限，避免二階循環**

這套機制本身有被反覆觸發的風險——如果 Aggregator 每次都用同樣的轉向邏輯重新發問，而下游模型每次都卡進類似的循環，會產生一個**更高層級的循環**：Aggregator 和 specialist 節點之間的重試循環。

務實的做法，是設定明確的重試次數上限（例如 2-3 次），超過就切換策略（換 profile、降 temperature、或直接交給 Aggregator 自己給出保守回答），而不是無限重試。

## 第五章：成本的代價——「智能 Aggregator」不是免費的午餐

這套「截斷 → 判斷 → 轉問句 → 重試」的機制，聽起來很完整，但它有一個不能忽視的代價：**Aggregator 從一個「轉發者」，變成了一個「判斷者」**，而判斷本身需要消耗算力和時間。

### 5.1 增加的智能需求

原本如果 Aggregator 只是路由 / 彙整角色，現在它需要做：

- 判斷輸出是否卡循環（n-gram 重複偵測，至少要跑一次文本分析）
- 判斷殘缺輸出有沒有提取價值（語意層判斷，可能本身就需要呼叫一次 LLM 做摘要）
- 構造新的轉向 prompt（這本身是一次生成任務）
- 追蹤重試次數、決定何時放棄

如果這幾步都要用 LLM 來做判斷（尤其是「值不值得提取」這種語意層判斷），等於**Aggregator 自己也要跑推理**，而不只是轉發資料。對於一個原本設計成「Aggregator 無本地推理、只做雲端代理協調」的節點而言，這個需求變化可能要求重新調整整個節點的角色定位。

### 5.2 增加的反應時間

原本一次請求 = 一次推理延遲。現在變成：

```
第一次推理（卡循環，跑到 num_predict 上限才停）
  + 偵測與判斷時間
  + 構造新 prompt 時間
  + 第二次推理（重新跑一次完整生成）
  + ……可能還有第三次
```

如果 `num_predict` 設得保守（比如 1500–2000 token 才觸發截斷），光是「跑到觸發截斷」這一步，就已經吃掉相當長的時間——等於讓系統先浪費一次完整的推理預算，才開始第二輪嘗試。

對於低延遲場景（例如即時語音助理、VTuber 互動），這個總延遲可能完全不可接受。**這正是這套機制最大的張力：用延遲和複雜度，換來穩健性。**

### 5.3 三個務實的折衷方向

**折衷一：把判斷邏輯做成輕量、非 LLM 的**

循環偵測本身（n-gram 重複比例）可以是純規則式的字串處理，不需要呼叫任何模型——這部分幾乎零延遲成本，應該保留。真正貴的是「摘要殘缺輸出」和「構造新問句」這兩步，如果能用簡單的模板規則取代（直接擷取截斷文本的前 N 個字當線索，而不是用 LLM 重新摘要），可以省掉一次額外的 LLM 呼叫。

**折衷二：提早偵測，而不是等到 num_predict 上限**

如果循環偵測能在生成過程中即時跑（stream 過程中每隔 N 個 token 檢查一次重複率），一旦發現循環苗頭就立刻中止——比如在 300 token 就抓到，而不是等到 2000 token 才觸發截斷，可以省下大部分浪費的生成時間。這對延遲的改善，通常比優化「截斷後怎麼辦」更直接有效。

**折衷三：接受「不是每次都要救」**

對即時互動場景（如語音助理、VTuber）而言，與其讓 Aggregator 做精細判斷，可能更實際的策略是：卡循環就直接用一句保底回應（「讓我換個角度想一下」、「這個問題我需要再確認一下」），把「重新生成完整答案」這個動作挪到背景非同步進行，前台先用一句安全的話撐住對話節奏——使用者感知到的延遲不會暴增，而真正的重試在背後悄悄做完，下一輪對話再把結果帶出來。

對於需要即時互動的應用（語音對話、VTuber 橋接），我會傾向先用這個策略——讓即時互動層保持輕量反應，把「判斷要不要重跑」這種重的邏輯放到背景非同步流程裡，而不是塞進對話的關鍵路徑上。

## 本篇小結

這一篇，我們從一份有錯誤的 Ollama Modelfile 建議開始，逐步拆解出：

1. **Ollama 參數命名的事實核查**——細節層面的準確性，不能假設 AI 生成的建議天然正確。
2. **兩種「卡住」的本質差異**——Transformer 的機率塌陷，vs Thinking 模式 stop token 訓練不足，vs（對照）液態神經網路的幾何極限環陷阱。三者表現相似，機制不同，解法也不同。
3. **截斷的真實侷限**——`num_predict` 解決的是運維問題（不要無限跑），不解決推理問題（給出有意義的答案）。
4. **截斷輸出的系統化再利用**——把殘缺輸出送進 Aggregator，當作構造新問句的線索，而不是直接丟棄。
5. **「智能 Aggregator」的代價**——增加的智能需求和反應時間，是這套機制不可忽視的成本，需要用務實的折衷來控制。

下一篇（Part II），我們將繼續深入：如何用「動態平衡」的策略表，讓不同模型在不同任務上找到各自的最適解；fine-tune 在這個架構中能解決什麼、不能解決什麼；以及 cross-model 協調為什麼注定是系統工程問題，而不是模型訓練問題。
